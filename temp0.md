
### Redis基础

Redis设计理念
* **基于内存** ：将**所有数据存储在内存中**。它的所有设计都围绕着**如何高效利用内存**、**如何减少内存碎片**的产生。
* **拒绝并发锁竞争的单线程模型：** **拒绝多线程模型**的线程切换和加锁解锁的昂贵开销。Redis**核心处理链路使用单线程**，屏蔽了多线程的复杂性。
* **把计算交给数据结构：** Redis 认为，**后端不应该拉取大量数据到业务应用里去排序或去重**。它直接在底层用 C 语言写好了最高效的底层数据结构

#### Redis架构模型


##### 单线程Reactor与Async Event

Redis 的核心网络模型基于 **I/O 多路复用**。
- 它并没有引入庞大的第三方网络库，而是封装了轻量级的 `ae` (Async Event) 事件驱动库。

**Async Event事件驱动库**
- 该库在**单线程**的 Event Loop事件循环中处理两类事件：
  * **File Events文件事件：** 处理客户端 Socket 的连接、读取（接收请求）和写入（返回响应）。
  * **Time Events时间事件：** 周期性执行的内部例程（如过期 Key 清理、字典 Rehash 步进）。
* **命令处理和 Lua 脚本执行，嵌套在文件事件处理的回调函数中**

**高并发业务开发约束：**
* **并发安全的基石：** 由于所有命令的执行（内存操作）都被串行化在一个 Event Loop 中排队处理，**绝对没有并发抢占内存数据的情况**。
  * 这就是为什么 Redisson 只需要通过原生命令和 Lua 脚本，就能构建绝对安全的分布式锁，无需关注底层锁竞争机制。
* **严格禁止 $O(N)$ 阻塞：** 
  * **单线程架构对 CPU 计算时间的容忍度极低**
  * 如果在业务中执行了如 `KEYS *`、针对包含数万元素的 `Hash` 执行 `HGETALL`，或者复杂的 Lua 脚本，该线程将被持续占用。
  * 在此期间，**Event Loop 无法处理任何其他 Socket 的就绪事件**，导致**整个微服务集群所有访问 Redis 的请求触发 `Timeout` 异常**，进而**引发连接池雪崩**。

##### C/S 架构、Lettuce、RESP协议

- Redis 基于标准的 Client-Server 架构，通过 TCP 长连接进行通信。在高并发场景下，客户端如何管理 TCP 连接决定了微服务节点的资源消耗。

- **Lettuce 相比 Jedis 的架构优势（Spring Boot 选型依据）：**
  * **Jedis（同步阻塞 I/O）：** 采用传统的线程池模式。一个 Java 业务线程必须独占一个 TCP 连接。在 1000 QPS 场景下，需要维护数百个 TCP 连接，导致 JVM 产生高昂的线程上下文切换（Context Switch）开销，且极易触发连接池耗尽异常。
  * **Lettuce（异步非阻塞 I/O）：** **基于 Netty 事件驱动框架实现。**它将请求封装为异步指令流，支持 **TCP 连接的多路复用**。这意味着数千个并发业务线程可以共享同一个或极少数几个底层的 TCP 连接，通过异步响应式回调获取结果。极大降低了微服务节点的内存占用和端口消耗。


**RESP 协议**REdis Serialization Protocol
- RESP 是 Redis 客户端与服务端通信的文本协议。
- 它设计极为精简，采用简单的边界标识符（如 `+`, `-`, `:`, `$`, `*`）和 CRLF (`\r\n`) 进行数据包的分割与解析。由于其解析复杂度仅为 $O(1)$ 或正比于负载长度，对 CPU 的解析压力极小。

**高并发业务开发约束**
* **网络带宽饱和瓶颈：** RESP 协议本身非常轻量，但**载荷的体积完全取决于客户端的序列化方式**。
  * Spring Cache 默认使用 JDK 序列化（`JdkSerializationRedisSerializer`），会附加大量的类元数据和版本信息，导致数据包急剧膨胀。
* **实战优化：** 
  * 在高并发读写下，膨胀的数据包会迅速打满内网千兆/万兆网卡的带宽上限。
  * 必须在 Spring Cache 配置中强制将 Value 的序列化器替换为 **JSON 序列化（如 `GenericJackson2JsonRedisSerializer`）** 或 **Protobuf**，以压缩 RESP 数据流的体积，降低 RTT（往返时延）。

##### 多线程IO

随着网络硬件的升级，Redis 的**性能瓶颈**从内存处理转移到了**网络 Socket 读写与协议解析**。
- Redis 6.0 引入了**I/O 多线程**，但**仅针对网络数据的读写和 RESP 协议解析**。
* **执行逻辑不变：** 
  * 主线程利用IO多路复用监听到请求
  * 主线程将就绪的 Socket 连接均匀地分配给IO 线程池，**期间主线程暂停执行**
  * 所有IO线程解析完命令后，主线程串行执行命令
  * 主线程将命令执行后的结果放在缓冲区，唤醒IO线程，IO 线程**并行**地将这些结果打包成网络字节流发回客户端
  * 主线程准备迎接下一个请求


**Redis 6.0 的多线程，仅仅是为了解决网络读写的瓶颈。它的核心命令和Lua 脚本执行，依然是单线程**
* Redisson 的加锁逻辑依然是线程安全的，Lua 脚本依然保持原子性。
* 但是：**多线程 I/O 无法解决大 Key 慢查询阻塞主线程的问题。**

##### 内存与持久化分离：RDB 与 AOF

为保证内存极速响应的同时不丢失数据，Redis 采用异步子进程机制进行持久化操作：
* **RDB（内存快照）：** 主线程调用 `fork()` 系统调用，利用操作系统的 Copy-On-Write (写时复制) 机制，由子进程将当前内存状态生成二进制快照文件。
* **AOF（追加日志）：** 以独立线程将写命令追加至日志文件，支持多种 `fsync` 刷盘策略（如每秒刷盘）。

**高并发业务开发约束：**
* **Fork 阻塞延迟：** 当 Redis 实例占用内存巨大（如大于 10GB）时，主线程执行 `fork()` 拷贝页表（Page Table）的操作会非常耗时，导致系统出现百毫秒级别的阻塞毛刺。
* **内存冗余评估：** 由于 Copy-On-Write 机制的存在，在持久化期间如果发生大量写操作（如高频的 Spring Cache 更新），操作系统会复制内存页，可能导致 Redis 实际占用内存飙升至原有的近两倍。需预留充足的机器物理内存（避免 OOM 导致进程被 Kill）。
* **实战策略：** 生产环境通常采用 **混合持久化（AOF 结合 RDB 快照）**，并建议在从节点（Slave）执行耗时的持久化操作，以保证主节点（Master）全力处理客户端的高并发请求。

##### 全局哈希字典


在 Redis 的单线程宇宙里，所有的键值对（不管你是 String、List 还是 Hash），最终都存放在一个全局的、极其庞大的数组里。这个核心结构在 C 语言源码中叫做 `dict`（字典）。

- **`dictEntry` 节点：** 这是全局数组中的每一个槽位（Bucket）。当你执行 `SET order:123 100` 时，Redis 实际上是向操作系统申请了一块极小的内存，创建了一个 `dictEntry` 结构体。
- **三根致命指针：** 每一个 `dictEntry` 内部都有三根物理指针，它们是串联整个 Redis 帝国的经络：
  - **`*key` 指针：** 永远指向一个 SDS（用来存你的键名，比如 `"order:123"`）。
  - **`*val` 指针：** 这就接上我们刚刚学的知识了！它指向的就是那个封装了具体业务数据的 **`redisObject`**（比如这里指向一个内部采用 `int` 编码的 `redisObject`）。
  - **`*next` 指针：** 用来解决哈希冲突的拉链法指针。

**宏观认知：** Redis 查数据极其狂暴的快，根本原因不是一句废话般的“因为它在内存里”，而是因为这个全局哈希表的寻址时间复杂度是 **O(1)**。CPU 通过 Hash 算法算出数组下标，瞬间顺着 `*val` 指针找到 `redisObject`，直接提取数据。没有任何多余的指令浪费。

---

**底层博弈：哈希冲突与“渐进式 Rehash”的续命狂奔**

随着微服务的并发写入，几千万甚至上亿个 Key 砸进 Redis，全局数组注定会爆满。这时候 `*next` 指针拉出的单向链表会越来越长，O(1) 的查询时间就会退化成 O(N)。单线程的 CPU 必须被迫顺着链表一个个对比 key，性能开始断崖式下跌。

- **被迫扩容（Rehash）：** 为了保命，Redis 必须申请一个比原来大一倍的全新全局数组。
- **致命的卡顿危机：** 如果让单线程把 5000 万个 `dictEntry` 一次性全部拷贝、重新计算 Hash 并挂载到新数组，这个过程可能需要几秒甚至十几秒。这期间单线程被彻底锁死，整个微服务集群的缓存请求会全部卡死在队列里！
- **物理级破解 —— 渐进式 Rehash：**
  - Redis 极其狡猾地同时在内存里保留 **两个哈希表**（`ht[0]` 和 `ht[1]`）。
  - 它把庞大的迁移工作量**碎片化**。之后微服务每一次对 Redis 执行增删改查时，单线程除了处理你的业务请求，还会“顺手牵羊”把 `ht[0]` 对应槽位里的几个元素迁移到 `ht[1]` 中。
  - **用极长的时间跨度，把致命的同步阻塞摊平为无感的异步微操作。** 这才是架构级的美学。

---

**架构师的梦魇：全局字典在微服务下的连环雷区**

理解了全局字典和 Rehash，你就能看透生产环境中那些足以让公司宕机的灵异事件。

- **陷阱 1：Rehash 期间的内存暴增与 OOM 击穿**
  - **物理推演：** 在渐进式 Rehash 的漫长周期内，Redis 内部同时存在两张巨大的哈希表。如果此时你的微服务遭遇双十一流量洪峰，疯狂写入新订单数据（新数据会直接写进新表 `ht[1]`），Redis 的物理内存占用可能会**瞬间翻倍**。
  - **灾难后果：** 如果宿主机的 `maxmemory` 阈值预留不足，Redis 会立刻触发内存淘汰策略（Eviction），疯狂删除线上正在用的热点 Key。微服务去查 Redis 查不到，海量的读并发瞬间击穿到底层 MySQL，数据库直接报废。
- **陷阱 2：`KEYS *` 命令的全局屠杀**
  - **物理推演：** `KEYS *` 的底层逻辑是强行挂起一切业务，命令单线程顺着全局字典的数组，从第 0 个槽位一直死磕遍历到最后一个槽位。
  - **灾难后果：** 只要有实习生在生产环境敲下这个命令，几千万个 `dictEntry` 会让单线程彻底停摆。微服务集群的超时重试风暴会在 3 秒内打满所有应用服务器的工作线程池。**架构铁律：生产环境必须通过 `rename-command` 物理禁用 `KEYS` 命令，强制要求开发使用带游标分批拉取的 `SCAN` 命令！**
