

## Redis

### Redis基础与核心概念
#### Redis技术定位与核心业务应用场景


##### Redis 的生态位与技术定位


**为什么必须引入 Redis？**
- 既然 MySQL 也能装在拥有 256GB 内存的服务器上，并且有 Buffer Pool 做内存缓存，为什么还要引入 Redis？答案隐藏在**底层存储介质的物理鸿沟**与**数据结构的执行引擎**之中。
*   **MySQL 的物理瓶颈（沉重的关系代数与磁盘 I/O）**
    *   **B+ 树**：MySQL 的核心设计哲学是 **“数据的绝对安全与 ACID 事务”**。这就决定了它的数据最终必须落盘（哪怕是 SSD）。在极高并发的读写下，MySQL 必须维护复杂的 B+ 树分裂、合并，以及庞大的 Undo/Redo Log 刷盘。
    *   **I/O 寻址的降维碾压**：即便 MySQL 命中了内存缓冲池（Buffer Pool），它的查询也必须经过 SQL 词法解析、优化器生成执行计划等沉重的链路。如果是磁盘读取，SSD 的随机读取延迟大约在 **100 微秒（$\mu s$）** 级别。当电商秒杀洪峰达到 10 万 QPS 时，这种微秒级的延迟会被瞬间放大，导致数据库连接池被打满，整个微服务集群雪崩。
*   **Redis 的内存高速公路（纯粹的内存寻址）**
    *   **RAM 的物理碾压**：Redis 是一款原生的 **In-Memory** 数据库。内存（RAM）的随机读取延迟在 **100 纳秒（$ns$）** 级别。这意味着 Redis 的物理硬件寻址速度，比基于磁盘的 MySQL 快了整整 **1000 倍**！
    *   **绕过文件系统**：Redis 的数据读写直接操作进程的堆内存空间，彻底绕过了 Linux 内核的 VFS（虚拟文件系统）和块设备驱动。它是微服务架构中真正意义上的“高速公路”。

--

**Redis 的生态位：数据结构计算服务器**
*   Redis 内置了 String、Hash、List、Set、ZSet 等极其丰富的数据结构，并针对这些结构提供了微操级别的 API（如 `HINCRBY`, `SADD`, `ZSCORE`）。
*   **计算向数据移动**：在 Redis 中，如果你要更新 `age`，只需发送一条极简的指令：`HINCRBY user:1001 age 1`。
*   **降维打击的本质**：Redis 将**计算逻辑直接下推到了服务端（内存所在地）**。微服务不需要拉取全量数据，只需发送一条指令。这彻底干掉了多余的网络 I/O，并且利用 Redis 单线程的特性，天然保证了微操的**绝对原子性**（无需加锁）。


> Redis 不是一个比 MySQL 快的缓存。
> 
> Redis 的真正定位是：**一个支持丰富数据结构、提供原子性微操的高性能内存计算与存储引擎。** 
> 
> MySQL 负责“保底与守序”（持久化、复杂事务、关系映射），而 Redis 负责“冲锋与破局”（抗并发洪峰、计数限流、轻量级队列）

##### Redis单线程为什么能扛住十万 QPS


**多线程一定快吗？**
- **在纯内存操作的场景下，多线程带来的副作用远大于它的收益**。
*   **1. 上下文切换的物理开销（纳秒与微秒的降维打击）**
    *   CPU 在切换线程时，必须保存当前线程的运行上下文（程序计数器、寄存器状态、调用栈指针），并加载下一个线程的上下文。更致命的是，这会导致 CPU 的 L1/L2 Cache 瞬间失效。
    *   **物理耗时对比**：一次 CPU 的上下文切换大约需要 **3 到 5 微秒（$\mu s$）**。而一次纯内存的读写操作仅仅需要 **100 纳秒（$ns$）** 左右。
    *   **为了让多线程去并发执行一个极快的内存操作，光是切换线程花费的时间，就足够单线程串行执行 30 到 50 次内存读取了！** 

*   **2. 数据结构与锁的无底洞**
    *   如果 Redis 是多线程的，当线程 A 在触发更新时，线程 B 又来读取数据，系统**必须引入重量级的并发锁**。
    *   加锁会导致线程频繁挂起和唤醒，解锁会导致总线风暴。最终，多线程非但没有提高并发，反而会让 CPU 把绝大部分算力耗费在“抢锁”和“排队”上。
    *   单线程天然屏蔽了并发冲突，让极其复杂的数据结构操作实现了绝对的 **无锁化（Lock-free）** 和 **原子性**。

> Redis 所谓的单线程，指的是**网络 I/O 读写与命令执行**（Redis 6.0 之前）是单线程的。实际上，Redis 内部一直都有后台线程（BIO Thread）在默默干活，比如异步关闭文件描述符（close file）、AOF 磁盘刷盘（fsync）、以及异步删除超级大 Key（UNLINK），这些极度耗时的重 I/O 操作都被剥离到了主线程之外。

---

**为什么Redis 能在普通的廉价服务器上跑出 10 万+ 的 QPS？**
1. 纯内存操作
2. 核心流程的单线程执行
   - 纯内存的串行执行非常快，能跟上网卡接收数据的速度
   - 单线程不仅省去了锁的开销，也极大降低了 Redis 底层 C 源码的开发和维护成本。
3. I/O 多路复用机制：Reactor单线程模型

##### 旁路缓存与双写一致性


**Redis 为什么是数据库防弹衣？**
- 在宏观的微服务架构中，由于磁盘 I/O 的物理限制，MySQL 往往是整个系统中最脆弱的环节。
- 一旦未经任何拦截的高并发读请求直接打到数据库，MySQL 会在几秒钟内因为连接池耗尽、CPU 打满而彻底宕机。
- Redis通过**旁路缓存模式**充当阻挡流量洪峰的防弹衣



**Cache-Aside 旁路缓存的标准工作流：**
*   **读请求**：微服务先查 Redis。命中则直接返回；未命中则去查 MySQL，查到数据后将其写回 Redis，并返回给前端。
*   **写请求（核心争议区）**：微服务先更新 MySQL，然后**删除** Redis 中的旧数据。

---

**为什么先更新数据库，再更新缓存是错的？为什么必须先更新数据库，再删除缓存？**
- 更新数据库 -> 更新缓存策略
  1. 线程 A 将数据库的值更新为 1。
  2. 线程 B 将数据库的值更新为 2。
  3. 由于网络抖动或操作系统的线程调度，**线程 B 先于线程 A 覆盖了 Redis 缓存**。
  4. 线程 B 将缓存更新为 2，随后线程 A 姗姗来迟，将缓存更新为 1。
  **致命后果**：此时数据库里的最新值是 `2`，但 Redis 里的值却被覆盖成了 `1`。两边数据产生了长时间的脏数据，除非等到该 Key 自动过期，否则微服务将一直读取到错误的数据。

- **运算成本与懒加载**
  - 在真实的业务中，存入 Redis 的往往是经过微服务**多表 Join、复杂计算后生成的一个庞大的 JSON 视图**
  - 如果每次修改一个底层字段，都要重新计算一次并更新缓存，那 1 分钟内如果有 100 次修改，就要执行 100 次复杂的视图计算。
  *   但如果在这一分钟内，压根没有任何用户来“读取”这个数据呢？这 100 次计算和网络 I/O 就全浪费了。
  *   **删缓存的智慧**：采用【删缓存】策略，本质上是一种**懒加载**的思想。我不去管你怎么算，我只负责把缓存清掉。只有当下一个读请求真正到来时，微服务才会去触发那一次沉重的计算并放入缓存。

> **双写一致性**
> 
> 先更新 DB，再删除 Cache已经是业界的通用标准。但它依然有一个极小概率的并发漏洞
> 
> 读请求在缓存失效的瞬间去读 DB（拿到旧值），此时另一个写请求更新了 DB 并删除了缓存，然后读请求才把刚才拿到的旧值写回缓存。
> 
> 解决这个极小概率问题的终极方案不是在业务代码里加复杂的分布式锁，而是**引入中间件监听 MySQL 的 Binlog，异步地去执行 Redis 的删除/更新操作**。这是架构解耦的最高境界。

##### 无状态微服务与分布式 Session

Redis 的第二重身份：**微服务状态的锚点**。


在单体架构时代，用户的登录状态（Session）是保存在 Tomcat 所在 JVM 的本地内存里的。
当架构演进为微服务集群时，灾难发生了：
1. 用户向网关发起登录请求，负载均衡器（Nginx）将请求路由到了 **节点 A**。节点 A 验证密码成功，在本地内存存下 Session。
2. 用户接着点击“我的购物车”，负载均衡器将请求路由到了 **节点 B**。
3. 节点 B 去自己的本地内存里找，发现没有这个用户的 Session，直接返回 `401 Unauthorized` 强制用户重新登录。

**无状态计算与 Redis 状态集中化**
- 为了让微服务集群能够随时随地扩容（比如双十一瞬间从 100 台机器扩容到 1000 台），微服务本身必须是**完全无状态（Stateless）**的。即：节点不能在本地 JVM 内存里保存任何会影响后续请求的上下文信息。
- 所有的状态，必须被剥离出来，交给一个全局统一的、极速的“状态维系者”来管理，它就是 Redis。

**分布式 Session 的宏观工作流：**
1. 用户在任意一个节点登录成功，微服务生成一个唯一的 `Token`（如 UUID），并将 `Token` 作为 Key，用户的身份信息作为 Value，存入 Redis，并设置过期时间（如 30 分钟）。
2. 微服务将这个 `Token` 返回给前端（写入 Cookie 或 Authorization Header）。
3. 用户发起后续请求，携带 `Token`。无论请求打到节点 A、B 还是 Z，微服务拦截器统统拿着这个 `Token` 去 Redis 里执行一次 `GET token` 操作。
4. 只要能从 Redis 读到数据，就证明用户处于登录状态。且每次读写操作，微服务都可以通过 `EXPIRE` 命令为这个 Session 自动续期。


> 在容器化（K8s）时代，微服务的 Pod 就像是流水线上的工人（随时可以被销毁和替换，因为它们是无状态的）；
> 
> 而 Redis 就是那块记录着所有流水线进度的全局黑板。没有 Redis，横向扩展（Scale Out）将是一纸空谈。

##### 分布式锁

在单体架构时代，当多个线程试图同时扣减同一件商品的库存时，我们只需在业务方法上加一个 `synchronized` 或 `ReentrantLock` 就能完美解决。这是因为所有的线程都跑在**同一个 JVM 进程**的内存里。

当系统演进为微服务集群，订单服务被部署在 10 台物理机上。此时，A 机器的 JVM 和 B 机器的 JVM 犹如两座孤岛，它们根本不知道对方的锁状态。如果两台机器同时查到库存还有 1 件，同时执行扣减，就会引发极其严重的**超卖超发**。

此时，我们需要一个**独立于所有 JVM 之外的全局裁决者**，这就是 Redis 分布式锁的生态位。

**Redis 分布式锁的进化简史**

**1. 极简的 `SETNX`**
   * 利用 Redis 单线程串行执行的特性。谁能用 `SETNX lock_key 1`（Set if Not eXists）成功写入，谁就拿到了锁。执行完业务后，调用 `DEL lock_key` 释放锁。
   * 死锁：如果微服务 A 刚刚拿到锁，还没来得及执行 `DEL`，所在的物理机突然断电宕机了（或者由于 OOM 被操作系统强杀）。这个锁将永远留在 Redis 里，其他所有机器都将在无尽的等待中卡死。

**2. 引入过期时间TTL**
   * 给锁加上寿命。拿到锁的同时，设置一个过期时间（比如 10 秒）。哪怕微服务挂了，10 秒后 Redis 也会自动清理这个锁，打破死锁。
   * 并发越界：命令 `SET lock_key value NX EX 10`保证加锁的原子性，但是如果业务极度复杂，执行时间超过了持有锁的时间，线程A业务没执行完就被解锁，导致其他线程获取到锁，线程A解除其他线程的锁，导致并发秩序崩溃


3. Redisson 与看门狗Watchdog
   - **Redisson**在底层使用了复杂的 Lua 脚本，完美解决了分布式锁的终极痛点：
   *   **防误删机制（解铃还须系铃人）**：Redisson 加锁时，不会存无意义的 `1`，而是存入当前线程的唯一标识（UUID + ThreadId）。在执行 `DEL` 释放锁之前，它会先通过 Lua 脚本判断：“这把锁的标识是不是我自己的？”只有身份匹配，才允许删除。
   *   **看门狗机制（动态续命）**：如果业务到底要执行多久无法预估，过期时间设多少合适？Redisson 的智慧在于：它默认给锁设置 30 秒的 TTL，但同时会在后台启动一个**看门狗线程（Watchdog Thread）**。
       *   只要持有锁的业务线程还在正常运行，看门狗就会每隔 10 秒（TTL / 3）去 Redis 里跑一次 Lua 脚本，把锁的过期时间**重新拨回 30 秒**。
       *   如果微服务物理机宕机了，看门狗线程也随之死亡，不再续期。30 秒后，Redis 自动释放锁。

##### 接口限流防刷

在真实的公网环境中，如果没有限流机制，恶意爬虫、黑客暴力破解、或者突发的营销活动洪峰，瞬间就能将网关和数据库击穿。Redis 凭借其极高的内存计算速度，成为了全网公认的最佳限流器。

大厂经典的限流方案通常有两种架构选型：

**1. 简单粗暴的“固定窗口计数器”（基于纯 String）**
*   **场景映射**：限制某个 IP 地址每分钟最多只能调用 100 次短信发送接口。
*   **底层指令**：`INCR ip:192.168.1.1` + `EXPIRE 60`。
*   **架构推演**：利用 `INCR` 的原子性累加。请求来了就 +1，如果发现返回值大于 100，直接返回 `HTTP 429 Too Many Requests`。
*   **致命缺陷（临界点突刺）**：假设恶意用户在第 59 秒疯狂发送 100 个请求，在第 61 秒（下一个计费周期的第 1 秒）又发送了 100 个请求。从 Redis 的逻辑来看，两个 60 秒的窗口都没有超标，但实际上在短短的 2 秒内，系统承受了 200 次请求的洪峰！这就是经典的“临界点雪崩效应”。

**2. 绝对严谨的“滑动窗口限流”（基于ZSet）**
- 为了解决临界点突刺，我们必须引入时间坐标系，而 Redis 的 ZSet（有序集合）天生就是干这个的。
*   **架构思路**：将每次请求记录在 ZSet 中。`Score`（分数）存入当前请求的毫秒级时间戳，`Member`（成员）存入随机 UUID（防止并发请求的精确时间戳重复导致数据被覆盖）。
*   **宏观工作流（Lua 脚本封装包装原子性）**：
    1.  微服务收到请求。
    2.  利用 `ZREMRANGEBYSCORE` 斩断历史：删除 ZSet 中所有时间戳小于 `当前时间 - 60秒` 的数据。（把 1 分钟以前的过期数据踢出窗口）。
    3.  利用 `ZCOUNT` 统计当前窗口内的元素个数：如果个数 $< 100$，放行；如果 $\ge 100$，拒绝。
    4.  如果放行，利用 `ZADD` 将当前请求打入 ZSet。
*   **架构选型博弈**：ZSet 滑动窗口的精度是完美的，它彻底消灭了临界点突刺。但代价是**极大的内存开销**（需要记录这 1 分钟内的每一次请求明细）和相对复杂的 $O(\log N)$ 计算。在绝大多数非金融级的业务中，固定窗口已经够用；只有在对流量把控极度苛刻的核心路由网关，才会上 ZSet 滑动窗口。

### 核心数据结构与底层内存实现机制



#### 统一对象模型redisObject的装箱开销与指针迷宫

为了实现 C 语言中的**多态**，即**用同一套 API 处理 String、List、Hash 等不同底层结构**，Redis 被迫给所有真实数据套上了一层**统一的物理外壳**。在海量小 Key 的场景下，这个外壳将化身为最恐怖的“内存刺客”。

##### 16 字节的物理装箱

在 C 语言这种极度贴近硬件的语言中，本没有对象的概念。为了让同一套操作指令（比如 `GET`）能智能识别并处理不同的底层数据结构，Redis 强行在所有真实数据外面，套了一个 **16 字节（128 bits）** 的**统一物理头：`RedisObject`**。

我们将这 16 字节逐个 bit 劈开：

*   **`type` (4 bits) —— 逻辑伪装**：向外部客户端声明它是谁（String、List、Hash、Set、ZSet）。这是给“人”看的。
*   **`encoding` (4 bits) —— 物理路由**：向底层 CPU 声明它实际的物理存储格式（`int`、`raw`、`embstr`、`ziplist`、`hashtable` 等）。这是给“机器”看的。
    *   **多态路由的物理实现**：当客户端发起操作时，Redis 首先读取前 4 bits (`type`) 校验命令是否合法；校验通过后，读取紧接着的 4 bits (`encoding`)，底层 C 代码通过一段 `switch-case` 路由表，精准跳转到对应的物理内存处理函数。这就是多态的本质。
*   **`lru` (24 bits) —— 生死时钟**：记录该对象最后一次被访问的时间戳或访问频率，专门留给内存淘汰算法（LRU/LFU）做生死裁决。
    *   *(至此：4 + 4 + 24 = 32 bits，刚好榨干第一个 **4 字节**)*
*   **`refcount` (4 bytes) —— 内存防线**：引用计数器。当有其他数据结构复用它时加 1，被删除时减 1。当跌到 0 时，触发 `jemalloc` 物理回收。
*   **`ptr` (8 bytes) —— 寻址迷宫**：一个 64 位的真实内存指针。顺着这个地址找过去，才是真正存放数据的有效载荷（Payload）。
    *   *(合计：4 + 4 + 8 = **16 字节**)*



**二次寻址与 CPU Cache Miss 灾难**
- 在现代计算机体系中，CPU 绝不会一个字节一个字节地读内存。它每次去 RAM（主存）捞数据，都是按 **Cache Line（通常 64 字节）** 连锅端，塞进自己的 L1/L2 超高速缓存中。
*   **缓存的落空**：当 Redis 执行读取时，CPU 瞬间将 16 字节的 `robj` 拉入 L1 缓存。此时 CPU 准备读取真实数据，但它绝望地发现，真正的数据并不在这里。
*   **指针的迷宫**：CPU 只能顺着 `robj` 里那个 8 字节的 `ptr` 指针，去寻找真实载荷。但真实数据是在 `jemalloc` 另外独立分配的物理页上，与 `robj` 在物理内存地址上**毫无关联、完全不连续**。
*   **物理后果（降维打击）**：CPU 在 L1/L2 中找不到 `ptr` 指向的遥远地址，触发极其致命的 **Cache Miss（缓存未命中）**。原本只需 1 纳秒的高速读取瞬间失效，CPU 被迫停工（Stall），耗费上百个时钟周期去慢速的 RAM 主存里进行**二次寻址（Pointer Chasing）**。在 Redis 单线程百万并发的压榨下，这种物理级的中断是抹杀吞吐量的绝对黑洞。


**为什么不用Tagged Pointer？**
- Tagged Pointer：在指针处直接塞long 型整数或 7 字节以内的字符串
- 主要原因：
  - 如果 ptr 有时是一个真实的内存地址，有时是真实数据，Redis 所有的底层函数在每次处理前都必须加一个逻辑分支，这种类型检查在每秒十万次的请求压力下，会带来大量的**CPU 分支预测失败**，严重拉低吞吐量。
- 但是Redis采用了其他两种优化方案，接下来的内容会讲：int 编码与 embstr

**万物皆对象的“装箱税”**
- 这 16 字节不是可选配置，而是**强行绑定的物理原罪**。
- 只要你向 Redis 写入一个数据，哪怕你的 Value 只有一个字符 "1"（真实载荷 1 字节），Redis 也必须先向 OS 申请这 16 字节的元数据头。

*   **物理成本的绝对刚性**：在海量小 Key 的生产场景下（如设备状态机、简单的布尔值打标），这 16 字节的 Overhead（额外开销）是无法逃避的物理税。你花重金买来的 256GB 物理服务器内存，可能有一大半全交了这笔“装箱税”，根本没用来存真实业务数据。




**94% 元数据的“内存倒挂”**

*   **极端的物理错位**：假设你的业务是海量设备状态机打标，Value 只是一个简单的字符 `"1"` 或 `"0"`（真实物理占用 **1 字节**）。
*   **装箱的绞肉机**：为了存这 1 字节的真实数据，你必须先向系统上缴 16 字节的 `robj` 装箱税。
*   **量化惨剧**：有效载荷 1 字节，元数据 16 字节。**有效数据占比仅 5.8%，而用于维持多态的元数据开销（Overhead）高达 94.2%！**
*   **宏观崩塌**：当你往集群里打入 1000 万个这样的小 Key 时，你花了几个 G 的昂贵物理内存，结果 94% 全用来存那个引发 Cache Miss 的“包装盒”。这就叫极度荒谬的“内存倒挂”。

##### 0-9999 共享池

既然每次新建 `robj` 都要缴纳 16 字节的装箱税，而业务场景中又充斥着大量的极小整数（如状态码 0、1、2），Redis 决定在物理层面进行一次“作弊”。
*   **物理预热（捏造池子）**：在 Redis 进程刚启动，还没接收任何客户端请求时，它会在内存中硬核循环 10000 次，强行预先分配好 10000 个完整的 `robj` 结构体，它们内部的 `ptr` 分别指向物理整数 0 到 9999。这就是**共享对象池**。
*   **指针的偷梁换柱**：当你的业务执行 `SET user:status 1` 时，Redis 发现真实载荷是数字 `1`。它**绝对不会**去调用 `jemalloc` 申请新内存。它直接让字典树上的 Value 指针，**物理重定向**到刚才预热好的那个代表 `1` 的共享 `robj` 内存地址上。
*   **物理结果**：仅仅做了一次指针赋值（8 字节），真实 Value 的 **16 字节装箱税被彻底抹除**。如果你有一千万个 Key 的值都是 `1`，它们在底层全部指向上帝视角的同一个物理地址，靠 `refcount`（引用计数）维持生命。
*  **访问链条：持有键->查Redis全局哈希字典->共享池->拿真实数据**


**LRU 时钟冲突与共享池的瞬间报废**
- 这个物理外挂看似完美，但它与分布式缓存最核心的淘汰机制——**LRU（最近最少使用）** 存在着不可调和的底层物理矛盾。

*   **物理时钟的抢占**：还记得前一节解剖的 16 字节吗？里面有一个 24 bits 的 `lru` 字段。为了实现 LRU 淘汰，**每次一个 Key 被读取，Redis 都会更新这个 `lru` 字段的物理时间戳。**
*   **并发写入的物理灾难**：如果一千万个 Key 共享同一个代表 `1` 的 `robj`，当客户端访问了 `Key A` 时，底层去更新了共享对象的 `lru` 时间戳。那么问题来了——`Key B`明明半年没被访问过，但因为它和 `Key A` 共用同一个物理身体，它的“最后访问时间”也被强制刷新了！
*   **LRU 算法彻底失效**：物理时钟的污染，导致 Redis 根本无法分辨这一千万个 Key 到底谁才是真正的“冷数据”。
*   **雪崩陷阱（一剑封喉的源码设定）**：为了保证 LRU 算法的正确性，Redis 源码中有一道冷酷的防御机制：**只要你在 `maxmemory-policy` 中开启了任何带有 LRU 或 LFU 的淘汰策略（如 `allkeys-lru`），Redis 会在启动时直接将这 10000 个共享对象池彻底禁用、当场报废！**
*   **线上惨案**：如果你在生产环境，为了防止内存写满，通过 `config set` 动态将淘汰策略从 `noeviction` 改成了 `allkeys-lru`。原本靠共享池活下来的一千万个小 Key，瞬间失去了物理依托。Redis 被迫在一瞬间为它们每人补发 16 字节的装箱税，**物理内存瞬间暴涨 160MB+**，极易直接击穿物理机内存上限，触发 OOM 绞杀。

#### SDS

**C 原生字符串的物理原罪与二进制安全**
- `\0` 截断引发的物理灾难
  - 在 C 语言物理内存结构中，原生字符串被定义为一组以空字符 `\0`结尾的字节序列。
  *   **截断黑洞**：OS 与 C 标准库在读取数据时，物理指针会从起始地址向后扫描，直到撞上第一个 `\0` 即停止。
  *   **二进制安全溃败**：当你尝试在 Redis 中存储一张图片、一段经过 Protobuf 压缩的二进制流时，这些数据内部高频率出现物理字节 `00`。
  *   **物理后果**：C 原生逻辑会误认为这是字符串终点，导致 `00` 之后的数据在物理层面上被“永久隐身”。对于数据库而言，这就是致命的**数据截断与损坏**。
  *   **SDS 的刚性防御**：SDS 废弃了以 `\0` 判断结尾的逻辑，在物理 Header 中引入了 `len` 字段。Redis 读取数据时不再“盲目探测”，而是根据 `len` 记录的物理长度进行**精确位移读取**，从而实现了对图片、音频、压缩流的二进制安全（Binary Safe）支持。



- **从 O(N) 性能退化到内存覆盖**
  - C-String 在内存操作上的原始性，是高并发环境下的性能天敌与安全隐患。
  *   **`strcat` 的性能绞肉机**：
      *   **物理动作**：在 C 语言中执行 `strcat(dest, src)`，系统必须先从 `dest` 的起始地址开始物理扫码，直到找到结尾的 `\0`（时间复杂度 O(N)）。
      *   **性能退化**：如果你在 Redis 中频繁追加数据（APPEND），随着字符串增长，扫描的时间开销会线性膨胀。
      *   **SDS 的降维打击**：SDS 通过 `len` 直接定位结尾，追加操作的起始寻址开销恒定为 **O(1)**。
  *   **缓冲区溢出**：
      *   **物理原罪**：C 原生 `strcat` 默认假设目标内存块（dest）拥有足够的承载空间，直接强行拷贝数据。
      *   **内存覆盖**：如果程序员没有手动计算并申请空间，写入的数据会物理溢出到相邻的内存区域，强行覆盖掉属于其他 Key 的元数据或指针，导致整个 Redis 进程发生不可预知的物理崩溃或数据错乱。
      *   **SDS 的防御逻辑**：所有修改操作都会先对比 `alloc - len`。如果物理余量不足，SDS 会先进行**自动扩容**，确保存储边界的物理绝对安全。

**Header与数据的内存布局**

- 为什么要把 Header 拆成五份？
  - 在早期的 Redis 版本中，SDS 只有一种通用的 Header 结构。无论你存的是 10 字节还是 10GB 的数据，`len`（长度）和 `alloc`（已分配空间）都必须使用 `uint32_t` 或 `uint64_t` 来存储。
  *   **物理浪费的量化**：如果你存一个 10 字节的短字符串，却要用32 字节的变量来记录长度信息。**Header 的开销比数据本身还大**，这在海量小 Key 场景下是绝对的内存自杀。
  *   **按需头分配策略**：从 Redis 5.0 开始，SDS 根据字符串的长度，物理拆分成了 5 种 Header
      *   **`sdshdr5`**：针对 2 的 5 次方（32 字节）以内的超短字符串。它甚至没有 `alloc`，直接**把`len`字段塞进 `flags` 的高五位，低3位存储类型**。
      *   **`sdshdr8` / `16` / `32` / `64`**：分别使用 `uint8_t`、`uint16_t`、`uint32_t`、`uint64_t` 来存储 `len` 和 `alloc
      * 为什么是8的倍数？**现代 CPU 和操作系统读取内存的最小物理单位是 1 个字节**，即使8位数据的长度（最大8字节）用四位1000就能表示，但是CPU也要一次性读取1个字节，8位数据的长度还不如直接用`sdshdr8`存储
  *   **物理结果**：当存储 10 字节数据时，Header 仅占用 3 字节

**寻址逻辑**

* 每个 SDS Header 的末尾都有一个 **1 字节** 的 `flags` 字段。
*  **3 bits 定生死**：`flags` 字段的低 3 位被用来存储 **SDS 类型标识**（共 5 种类型，3 bits 刚好覆盖）。
* **后脑勺寻址法**：
    *   当 Redis 拿到数据指针 `ptr` 时，它直接指向数据 buf[0] 的位置，它会直接将指针**向前偏移 1 个字节**读取 `flags`，根据该字节的低三位判断类型
    *   如果读取到的是 `sdshdr8` 类型，它就知道再往前挪 2 个字节就是 `len` 
    *   如果读取到的是 `sdshdr16` 类型，它就知道再往前挪 4 个字节才是`len`

**内存对齐**与**对齐填充**
- **Cpu从主存读数据时，一次读8字节，即1Word字**
  - **对齐状态**：如果一个小于等于4字节的变量正好起始于 4 的倍数地址，CPU 只需一次内存访问就能把它完整取出。
  - **非对齐状态**：如果变量跨越了两个“字”的边界，CPU 必须执行两次内存访问，然后再通过复杂的移位、拼接操作把数据拼出来。这会导致性能损耗
- 按照 C 语言的默认规则，结构体会为了 CPU 读取效率进行**对齐填充（Padding）**。比如一个 `uint8_t` 后面跟着一个 `uint16_t`，编译器会自动在中间插入 1 字节的空洞
*   **物理压榨**：Redis 所有的 SDS Header 都**禁止任何对齐填充**，所有字段必须头尾相接，严丝合缝地挤在一起。
*   因为 Redis 是**内存密集型应用**。数亿个 Key 如果每个都多出 1 字节的空洞，叠加起来就是数百兆的物理内存浪费，这比微小的 CPU 周期损耗昂贵得多。

**alloc字段：预分配策略与惰性释放**
- 当字符串发生 `APPEND` 等增长操作时，SDS 不会卑微地仅向 OS 申请刚好够用的空间，而是采取“空间预分配”策略，试图通过物理冗余来彻底消灭频繁的 `realloc` 系统调用

- **扩容物理流水线：从 100 字节到 2MB**
  *   **阶段一（翻倍增长）**：假设当前 `len` 为 100 字节，追加后 `newlen` 为 500 字节。由于 `newlen < 1MB`，Redis 会申请 `500 * 2 = 1000` 字节的 `alloc` 空间。此时，物理内存中多出了 500 字节的空闲区（Unused）。
  *   **阶段二（线性增长）**：当追加导致 `newlen` 达到 2MB 时。由于 `newlen >= 1MB`，Redis 停止翻倍，改为申请 `2MB + 1MB = 3MB` 的 `alloc` 空间。
  *   **物理收益**：在下一次追加发生时，只要总长度未突破 `alloc` 边界，Redis 仅需修改 `len` 字段并移动指针，无需触发昂贵的 `realloc` 内存重新分配与数据拷贝，操作复杂度稳坐 **O(1)**。


**惰性释放与内存不收缩**
- 与扩容时的贪婪相反，SDS 在缩容时表现得极其懒惰。这种为了保护 CPU 响应速度的设计，往往是引发物理内存虚高的元凶。

*   **物理假象**：当你执行 `SET key "1"` 覆盖掉原本 10MB 的长字符串时，SDS 仅仅将 `len` 修改为 1。
*   **空间黑洞**：底层的 `alloc` 依然维持在 10MB，那些被废弃的 9.99MB 空间并不会立即归还给 OS，而是静静躺在内存条里等待复用。这就是**惰性空间释放**。
*   **生产惨案**：
    *   **指标背离**：你会发现 `INFO memory` 中的逻辑内存 `used_memory` 骤降，但操作系统层面的 `used_memory_rss` 纹丝不动。
    *   **内存耗尽**：如果业务中存在大量“先存长文本后改短字符”的操作，Redis 实例会呈现出“逻辑空虚但物理饱和”的诡异状态，最终引发 OOM。

#### String

##### `embstr` 与 `raw`
**44 字节的短字符串与 CPU Cache Line**
*   **物理底盘**：现代主流 CPU 从内存读取数据进入 L1 缓存时，最小的物理搬运单位是一个 **Cache Line（缓存行），固定为 64 字节**。如果你的数据能刚好塞进这 64 字节，CPU 就能一次性毫无阻塞地把它“生吞”进高速缓存。
*   **物理切割**：
    1. 必须缴纳的基础装箱税：`RedisObject` 头 = **16 字节**。
    2. 为了实现 SDS 的 O(1) 长度获取，必须带上最轻量级的头部：`sdshdr8`（包含 len、alloc、flags）= **3 字节**。
    3. 为了向后兼容 C 语言原生库的 API，必须在末尾强行追加一个 `\0` = **1 字节**。
    4. **极限压榨**：64 字节（总容量） - 16 字节 - 3 字节 - 1 字节 = **44 字节**。
*   **物理定局**：当真实 Payload（有效载荷） $\le$ 44 字节时，它加上所有元数据，刚好填满一个 64 字节的 Cache Line。这就是 `embstr` 编码诞生的物理基石。


**`embstr` 与 `raw` 的物理鸿沟**

*   **`embstr`（连续体 / 单次 Malloc）**：
    *   **内存动作**：调用 1 次 `jemalloc`，直接在内存中圈出一块最大 64 字节的连续物理空间。`RedisObject` 和 `SDS` 像连体婴儿一样严丝合缝地缝合在一起。
    *   **算力收益**：CPU 寻址时，一次内存 I/O 直接将结构体和数据全部拉入 L1 Cache。0 内存碎片，0 Cache Miss。
*   **`raw`（分离体 / 两次 Malloc）**：
    *   当 Payload 达到 45 字节，64 字节的缓存行彻底装不下。
    *   **内存动作**：Redis 只能妥协。第 1 次 `malloc` 申请 16 字节存放 `RedisObject`；第 2 次 `malloc` 去物理内存的另一个遥远角落申请空间存放 `SDS`。
    *   **致命黑洞**：原本相连的结构被物理斩断，只能靠 `robj` 里的 `ptr` 指针相连。CPU 读取完对象头后，顺着指针去主存中找数据，触发跨缓存行的 **Cache Miss**。这不仅带来了两次系统调用的开销，更在单线程循环中引发了纳秒级的寻址阻塞。
*   **极致变异：`int` 编码**
    *   如果存的是纯数字（且在 8 字节能表示的整型范围内），Redis 连 SDS 都不要了！直接把数字硬塞进 `RedisObject` 那 8 字节的 `ptr` 指针内存空间里。0 字节多余开销，物理榨干。

##### 处理数字的三级判断

**Redis 在处理一个数字时，会按三级顺序进行判断**
- **第一级：共享对象池**
  - 当你执行 `SET key 100` 时，Redis 首先检查这个数是否在 `OBJ_SHARED_INTEGERS`（默认 0-9999）范围内。
  *   **操作**：Redis 不会为你创建新的 `RedisObject`。它直接让字典（Dict）里的指针指向那个预先捏造好的、全局唯一的物理地址。
  *   **内存成本**：
      *   **16 字节装箱税**：**0**（因为是全局共享的）。
      *   **指针成本**：8 字节（Dict 里的 Value 指针）。
  *   **结论**：这是最省钱的方案。无论一千万个 Key 的值是多少，只要在 0-9999 内，它们在物理内存里只占用一份 `RedisObject` 的空间。

- **第二级：`int` 编码**
  - 如果你执行 `SET key 123456`，这个数超出了共享池，但它依然是个整数。
  *   **操作**：Redis 会创建一个新的 `RedisObject`，将其 `encoding` 设置为 `OBJ_ENCODING_INT`。此时，`ptr` 字段不再存储内存地址，而是**直接存储数值 `123456` 本身**。
  *   **内存成本**：
      *   **16 字节装箱税**：**16 字节**（必须为这个 Key 建立独立的元数据，如 LRU、RefCount）。
      *   **SDS 成本**：**0**
  *   **结论**：虽然没能像共享池那样实现全局去重，但它消灭了 SDS 结构。相比存储字符串 "123456"，它**省去了申请 SDS 内存的开销和 SDS Header 的空间**。

- **第三级：SDS 编码（`embstr` 或 `raw`）**
  - 如果你执行 `SET key "999999999999999999999"`（超出了 8 字节长整型的范围），或者存的是 `"hello"`。
  *   **操作**：Redis 只能乖乖去调用 `jemalloc` 申请内存，创建 SDS 结构。
  *   **内存成本**：16 字节 (robj) + SDS Header + 实际数据长度 + `\0`。

| 场景 | 命中策略 | 是否有独立的 `robj` | 是否有 SDS | 内存占用（不计 Key） |
| :--- | :--- | :--- | :--- | :--- |
| **数值 0 ~ 9999** | **共享对象池** | 否 (全局共享一份) | 否 | 仅 8 字节指针 |
| **其他长整型** | **`int` 编码** | 是 (每个 Key 独立) | **否** | 16B (robj) + 8B (Value) |
| **字符串/超大整数** | **SDS 编码** | 是 | **是** | 16B + SDS Header + Data |

##### 只读`embstr`的降级陷阱

在 Redis 底层 C 源码中，`embstr` 编码的字符串实际上是**绝对只读（Read-Only）**的。
*   **物理绞死**：`embstr` 最大的优势是 `RedisObject` 和 `SDS` 通过一次 `malloc` 申请，生存在同一块物理内存中。这意味着如果要给它扩容（比如追加一个字符），底层必须对这整个一整块连续内存执行 `realloc`。
*   **代码的无情妥协**：为了避免处理连续内存重分配可能引发的极其复杂的指针偏移、空间拷贝以及潜在的内存越界问题，Redis 源码直接“偷懒”——彻底放弃了对 `embstr` 内存重分配的底层支持。
*   **粉碎重组**：当你对一个 `embstr` 执行 `APPEND` 动作时，Redis 绝不会在原地址尝试扩容。它会直接宣告这块连续内存“死亡”，强行在物理内存的另外两个遥远角落，分别执行两次 `malloc`，新建一个独立的 `robj` 和一个独立的 `SDS`，再把老数据完整拷贝过去。



**只要对 `embstr` 发起任何数据修改指令，降级就是瞬间且永久的**。

*   **降级鸿沟**：连续体被物理斩断，原本依靠一条 Cache Line（64 字节）就能完整载入 CPU 高速缓存的数据，瞬间被撕裂成了依靠 8 字节 `ptr` 指针跨页寻址的 **`raw` 编码**。
*   **绝对不可逆（单行道黑洞）**：这种物理降级是**不可逆的单行道**。一旦降级为 `raw`，哪怕你之后执行字符串截断操作，把长度重新缩减到 44 字节以内，它也**永远不可能**再复原成 `embstr`。
*   **分配开销翻倍**：原本极速的单次 `malloc` 申请，在修改的瞬间变成了昂贵的两次 `malloc`。如果你在生产环境中，把 String 当作高频修改的计数器或追加日志（且初始值是以字符串形式存入的），不仅会产生大量的内存碎片，还会彻底打烂原本完美的 L1 Cache 命中率。

##### String API使用规范


`INCR`
- 在秒杀扣库存、API 接口高频限流的战场上，普通的 `GET` 拿数据到本地加减，再 `SET` 回去，在并发下绝对会被打穿（Race Condition）。必须祭出 `INCR` / `DECR`。

*   **物理降维（免解析的暴力加法）**：
    当一个 String 的底层编码是 `int` 时（纯数字），真实数据就存放在 `RedisObject` 那 8 个字节的 `ptr` 指针槽位里。当你执行 `INCR` 时，Redis 底层**根本不会去进行任何字符串转数字的解析运算**。它直接把那个 8 字节的物理内存当成 C 语言的 `long` 类型，丢给 CPU 的 ALU（算术逻辑单元）执行一次硬件级别的 `++` 操作。
*   **绝对原子性**：
    配合 Redis 的单线程 Reactor 物理模型，这个 `++` 操作没有任何线程上下文切换，不需要任何系统级互斥锁（Mutex）。1 万个并发请求打过来，在底层网卡接收队列里物理排队，被单线程一个接一个地执行硬件累加。这就是秒杀场景下，`INCR` 能做到 10 万 QPS 且绝对不会超卖的物理底气。


`SETNX` 分布式锁的连环绞杀

- 利用 `SET key value NX EX 10` 来做分布式锁，是业务开发最高频的操作。但如果仅仅把它当成普通的 CRUD，系统迟早会被并发幽灵彻底撕碎。

*   **黑洞一：锁超时（业务执行 > TTL）的临界区重叠**
    *   **物理灾难**：线程 A 拿到了 10 秒的锁，但遭遇了 JVM 的 Full GC（STW）或者慢 SQL，阻塞了 12 秒。第 10 秒时，Redis 键物理过期。此时线程 B 趁虚而入拿到了锁。第 12 秒，线程 A 醒来继续执行业务。**此时，A 和 B 两个线程同时在临界区内修改同一行核心数据！**（防坑指南：必须引入 WatchDog 看门狗机制，后台线程定时给锁续期）。
*   **黑洞二：非原子性释放（误删别人的锁）**
    *   **致命绞杀**：承接上一步，线程 A 终于执行完了业务，习惯性地调用 `DEL key` 释放锁。但此时锁已经是 B 的了！A 直接把 B 的锁给物理删除了。紧接着，线程 C 长驱直入，和 B 一起在临界区里群魔乱舞。整个分布式锁体系瞬间雪崩。
*   **缝合手术：Lua 脚本的原子降维**
    *   **正确姿势**：释放锁时，必须判断 `Value` 是不是自己放进去的那个 UUID，是才能删除。但 `GET` 和 `DEL` 是两条独立的指令，中间仍然会被网络并发打断。
    *   **绝对物理防御**：必须用一段 Lua 脚本将这两条指令“缝合”起来：
      `if redis.call("get",KEYS[1]) == ARGV[1] then return redis.call("del",KEYS[1]) else return 0 end`
    *   在 Redis 底层，整个 Lua 脚本会被当做**一条绝对原子的单线程指令**推入执行引擎。在它执行期间，任何其他客户端的并发请求全部被物理隔离在外。这就叫用底层的绝对原子性，斩断网络层的并发黑洞。

#### Hash

##### Hash 核心命令与实战场景

**String(JSON) vs Hash**
- **“微服务开发中，遇到用户画像、商品详情这种属性众多的对象，到底是用 String 存成一个大 JSON，还是用 Hash 结构存储？”** 

> 1. 数据结构**扁平且需要高频修改部分字段**（如购物车、资产表），无脑上 **Hash**。
> 2. 数据结构 **嵌套极深，且业务总是“整存整取”**，不涉及局部微操（如订单快照、复杂配置项），使用 **String(JSON)** 序列化存储更简单。



*   **1. 局部更新能力与并发冲突（核心痛点）**
    *   **String(JSON)**：典型的“牵一发而动全身”。假设你要给用户增加 10 块钱余额，流程是：`GET` 取出完整 JSON -> 应用程序反序列化 -> 修改余额 -> 序列化 -> `SET` 写回。
        *   **致命缺陷**：在极高的并发下，传统的 `GET + SET` 会引发严重的**丢失更新（Race Condition）**。为了保证数据一致性，你不得不引入分布式锁（Redisson）或者乐观锁，这极大地拖慢了系统吞吐量。
    *   **Hash 结构**：天生支持字段级别的微操。
        *   **架构优势**：只需一条 `HINCRBY user:1001 balance 10` 命令。计算逻辑直接下推到 Redis 服务端，利用 Redis 的单线程模型完美实现了**绝对的原子性**，无需任何分布式锁，性能极高。

*   **2. 网络 I/O 开销与网卡带宽**
    *   **String(JSON)**：每次修改或读取哪怕只是一个 `age` 字段，都要在网络中传输整个 JSON 字符串。在千万级 QPS 的洪峰下，这种无谓的冗余数据传输会迅速打满服务器的物理网卡带宽，导致整个微服务集群响应超时。
    *   **Hash 结构**：按需索取。`HGET user:1001 age` 仅仅传输几十个字节的报文，将网络 I/O 压缩到了极致。

*   **3. 内存占用与物理碎片（承上启下）**
    *   **String(JSON)**：如果把对象的每个属性拆分成独立的 String key（如 `user:1001:name`, `user:1001:age`），根据你之前学过的知识，每个 key 都会产生一个独立的 `redisObject` 头信息和底层的 SDS 结构，带来可怕的**元数据指针开销**和**内存碎片**。
    *   **Hash 结构**：Redis 底层对小规模的 Hash 做了“变态级”的内存压缩机制（**Ziplist/Listpack**，下一节的重点）。它不仅省去了大量的指针开销，还通过紧凑的连续内存布局，极大地提升了 CPU Cache 的命中率。



---

**Hash 核心高频 API 清单**
- 在 Hash 的操作中，牢记一个口诀：“命令前缀都是 `H`，操作粒度细化到 `Field`”。

*   **写操作：**
    *   `HSET key field value [field value ...]`：设置一个或多个字段的值。（注意：Redis 4.0 后 `HMSET` 已被废弃，`HSET` 直接支持批量写入）。
    *   `HSETNX key field value`：仅当字段不存在时写入（常用于初始化对象属性，防并发覆盖）。
    *   **`HINCRBY key field increment`**：
        *   对某个整数字段进行原子加减（传负数就是减）。**高并发防超卖、计数器的绝对核心。**
        *   Redis 保证了如果不存在会自动初始化为 0 再加 1
*   **读操作：**
    *   `HGET key field`：获取指定字段的值。
    *   `HMGET key field1 field2`：批量获取多个指定字段的值。
    *   `HEXISTS key field`：判断某字段是否存在。
    *   `HLEN key`：获取 Hash 表中的字段总数。
*   **危险与遍历操作（避坑专用）：**
    *   **`HGETALL key`**：获取所有的 field 和 value。
        *   <font color="red">**高危预警**</font>：如果这个 Hash 里面存了 1 万个字段，执行 `HGETALL` 会导致 Redis 单线程**严重阻塞**。生产环境中，针对大 Hash，必须换用 `HSCAN` 游标命令进行渐进式迭代。

**电商购物车**

购物车是高并发电商系统最经典的业务。如果把购物车存进 MySQL，在大促期间数据库会被极其密集的“加购”动作瞬间击穿。用 Redis 的 Hash 结构可以完美承载。

**1. 架构映射关系**
*   **外层 Key**：业务标识 + 用户 ID。例如：`cart:user:8848`。
*   **内层 Field**：商品标识 SKU_ID。例如：`sku:10086`。
*   **内层 Value**：购买数量 Quantity，或者是包含勾选状态的精简 JSON串。我们以最经典的“存数量”为例。

**2. 核心业务动作与 API 映射**
*   **用户添加商品到购物车 / 增加商品数量**
    *   **场景**：用户在商品详情页点击了“+1”。
    *   **命令**：`HINCRBY cart:user:8848 sku:10086 1`
    *   **宏观认知**：不要先 GET 判断存不存在再 SET。直接用 `HINCRBY`，Redis 保证了如果 SKU 不存在会自动初始化为 0 再加 1，全程无锁，单核轻松抗下 10万 QPS。
*   **用户减少商品数量**
    *   **场景**：购物车列表页点击“-1”。
    *   **命令**：`HINCRBY cart:user:8848 sku:10086 -1`
    *   **注意**：应用层需要判断，如果返回值变成 0，立刻调用 `HDEL` 命令删掉这个 field。
*   **用户删除某件/批量删除商品**
    *   **命令**：`HDEL cart:user:8848 sku:10086 sku:10087`
*   **展示购物车总商品种类数（角标数字）**
    *   **场景**：App 底部的购物车 Icon 右上角的小红点数字。
    *   **命令**：`HLEN cart:user:8848`
*   **展示购物车清单页面**
    *   **场景**：用户点进购物车页面，需要渲染列表。
    *   **命令**：`HGETALL cart:user:8848` （在购物车场景下这是安全的，因为业务上通常会限制单个用户的购物车 SKU 上限，比如最多 100 种，不会造成 Redis 阻塞）。

##### Ziplist与级联更新


**为什么要设计 Ziplist？**

基于**对象的装箱开销** 与 **指针迷宫**来审视传统的哈希表，在存储极小数据时，有致命缺陷。
- 假设你要存储一个非常小的 Hash：`HSET user:1001 age 25`。如果直接使用标准的 C 语言结构（数组 + 链表）来做字典：
  1. **指针开销比数据还大**：在 64 位操作系统下，一个内存指针固定占用 **8 字节**。而你的数据 `age` 占 3 字节，`25` 占 2 字节。为了存 5 字节的数据，底层需要 `dictEntry` 的 `next` 指针（8 字节）、`key` 指针（8 字节）、`value` 指针（8 字节）……**元数据开销是实际数据的数倍！** 这在内存如金的 Redis 中是绝对不可接受的。
  2. **内存碎片的深渊**：传统的链式结构，节点是通过 `malloc` 动态分配在堆内存的各个角落的。这不仅会产生严重的物理内存碎片，更致命的是，它**彻底破坏了 CPU Cache 的空间局部性**。CPU 从内存读数据是按 Cache Line（通常 64 字节）读的，零散的节点会导致 CPU 缓存命中率极低。

**架构的救赎：Ziplist（压缩列表）**。
设计理念极度暴力且优雅：**抛弃一切内存指针，申请一整块极致紧凑的连续物理内存**。用“偏移量（Offset）”代替“指针（Pointer）”，通过精密的位运算来寻址。

---

**Ziplist 总体布局：连续内存**
- Ziplist 不是一个类，也不是一个普通的结构体，它是**一长串紧挨着的字节流**。完全靠entry的三个字段来逻辑划分每个节点的边界

```text
|<------------------------- Ziplist 占据的连续物理内存总大小 ------------------------->|

+---------+---------+-------+--------+---------+--------+---------+-------+
| zlbytes |  zltail | zllen | entry1 |  entry2 | entry3 |  entry4 | zlend |
+---------+---------+-------+--------+---------+--------+---------+-------+
  4 bytes   4 bytes  2 bytes  (动态)   (动态)    (动态)    (动态)   1 byte
                             [ key1 ] [ value1] [ key2 ] [ value2]
|<---- Ziplist Header ----->|<--------- 真实数据存放区 ---------->| 结尾标记
```

**核心字段工作职责拆解：**
*   **`zlbytes` (4 字节)**：记录整个 Ziplist 占用的物理内存总字节数。有了它，当系统需要重新分配内存（`realloc`）时，瞬间就能知道要拷贝多大的块。
*   **`zltail` (4 字节)**：记录**最后一个 entry** 距离起始地址的偏移量。
    *   *宏观认知*：这是实现 **O(1) 尾部弹出/访问** 的关键！无需遍历，拿首地址加上这个偏移量，指针直接飞到队尾。
*   **`zllen` (2 字节)**：记录 entry 的数量。注意，如果数量超过了 $2^{16}-2$ (65534)，这个字段就被塞满了，此时必须真正遍历一圈才能知道节点总数。
*   **`zlend` (1 字节)**：固定值 `0xFF` (十进制 255)，纯粹的标志位，告诉解析器：“踩到这就到底了，刹车！”。
*   **Hash 的存储映射**：在 Ziplist 中，Hash 的 Key 和 Value 是**紧挨着**作为相邻的两个 Entry 存入的。

---

**Entry 节点的寻址**

- **entry既可以存储key，也可以存储value，且每个entry的大小不固定，每个key-value的entry节点紧密相连**
- 指针初始化在起始位置，起始位置的头几个字节就是 prevlen。读取它后，用当前指针的地址减去这个 prevlen 的数值，指针就会精准跳回到上一个 entry 的起始位置。这就是 Ziplist 能够双向遍历的基础。


```text
+-----------------------+-------------------+-----------------------+
| prevlen               | encoding          | content               |
| (前一个节点的字节长度)| (当前数据的类型与长度)| (真实的业务数据)      |
+-----------------------+-------------------+-----------------------+
```

- **`prevlen`：逆向寻址的坐标**
  - 当指针指向entry的起始位置，往后读取一个字节就能知道prevlen占用1字节还是5字节
  - 如果前一个节点长度小于 254 字节，`prevlen` 占用 **1 个字节**
  - 如果前一个节点长度大于等于 254 字节，它会扩展为 **5 个字节**
    - 第一个字节固定 254 (0xFE)作为标志位，255 (0xFF)是Ziplist 的结束标志位zlend
    - 后 4 字节存真实长度

    | 前两位 (Bit) | 含义 | `encoding` 总长度 |
    | :--- | :--- | :--- |
    | `00` | 字符串，长度小于等于 63 字节 | **1 字节** |
    | `01` | 字符串，长度小于等于 16383 字节 | **2 字节** |
    | `10` | 字符串，长度大于 16383 字节 | **5 字节** |
    | `11` | **整数类型**（如 int16, int32 等） | **1 字节** |

-  **`encoding`：正向寻址与类型标记**
   - `prevlen`末尾再往后读取两位就能获取以下信息：
     - encoding占用的空间
     - content的类型
     - 如果content是字符串，则encoding还记录它的长度
     - 如果content是整数，还通过枚举映射整数类型
   - 如果是整数，则记录整数类型
     - 如果是极小的整数，Redis 甚至连 `content` 部分都省了，直接把数字“揉碎”硬塞进 `encoding` 这个字节的后几位里！

    | `encoding` 字节 (二进制) | 对应数据类型 | `content` 占用空间 | 整数值范围 |
    | :--- | :--- | :--- | :--- |
    | `11000000` | **int16** | 2 字节 | $-2^{15}$ ~ $2^{15}-1$ |
    | `11010000` | **int32** | 4 字节 | $-2^{31}$ ~ $2^{31}-1$ |
    | `11100000` | **int64** | **8 字节** | $-2^{63}$ ~ $2^{63}-1$ |
    | `11110000` | **24 位整数** | 3 字节 | $-2^{23}$ ~ $2^{23}-1$ |
    | `11111110` | **8 位整数** | 1 字节 | $-2^{7}$ ~ $2^{7}-1$ |

   - 知道encoding和content的占用空间，就知道下一个entry的起始地址


---

**Entry寻址：丢掉指针，用偏移量计算**
- 执行 `HGET user:1001 age`
- 获取这块连续内存的起始物理地址 `Base_Ptr`
- **跳过头信息**：`Base_Ptr + 10` (4+4+2) 字节，指针准确降落到 `entry1` (也就是第一个 Key) 的位置。
- **解析与比对**：读取 `entry1` 的 `encoding`，算出它的长度，取出内容与 "age" 比对。
- **往后跳跃**：如果不匹配，将当前指针位置 `+ prevlen占用字节` `+ encoding占用字节` `+ content大小`，指针瞬间“滑过”当前节点，精确降落到下一个节点。
- **命中目标**：找到 Key 为 "age" 的节点后，再用同样的偏移量加法往下跳一个节点，里面存的就是 Value！


---


Ziplist 用极其复杂的位运算和解析逻辑，换来了**近乎 100% 的内存利用率**和**极致的 CPU Cache 命中率**（因为内存是连续的，CPU 预读（Prefetch）机制会提前把后面的节点全部拉入 L1/L2 缓存）。这在微小数据规模下，是无敌的存在。

***

Ziplist 为了逆向遍历，在每个节点的头部放了一个 `prevlen`记录前一个节点的长度；
- 如果**前一个节点长度 < 254 字节** ，`prevlen` 只占 1字节 。
- 如果**前一个节点长度 >= 254 字节** ，`prevlen` 必须膨胀到 5 字节 。


**四种场景下的时间复杂度**
1. `prevlen` 占用空间不变，说明前一个节点的总长度没有超过253字节
   - 前一个节点的content长度和旧数据一样时，时间复杂度$O(1)$
   - 前一个节点的content长度变小会变大时，时间复杂度$O(N)$：从被修改的节点开始，所有节点必须整体向前或向后平移
2. `prevlen`占用空间变为5字节，说明前一个节点的总长度超过253字节，
   - 前一个节点的content长度变大，会导致后面的所有节点整体平移，时间复杂度为$O(N)$
   - 从被修改content的节点开始计数，第一个节点的prevlen膨胀为5字节，如果它的长度没有超过254，第二个节点的prevlent不需要改变，则时间复杂度为$O(N)$
   - 如果恰好处在倒霉的巧合之中，每个节点的长度**刚好都是 250 字节 - 253 字节**之间，则第一个prevlent的膨胀将导致它的长度超过253字节，级联导致下一个prevlent膨胀，引发连锁更新，**时间复杂度为$O(N^2)$**

>`prevlen`只允许膨胀，绝不允许缩小；通过浪费4字节避免有第二次级联更新


---


每次节点膨胀，Ziplist为了保持物理内存连续，都必须重新向操作系统申请一块更大的内存空间，然后把后面的所有数据做一次 `memcpy` 拷贝。
- content无论是缩小还是增大，整体都必须向前或向后迁移，这是固定的时间复杂度$O(N)$。
- 如果发生连锁更新，Redis 就要执行$N$次的内存重新分配和拷贝，退化成$O(N^2)$
- Redis 核心命令执行是单线程的，$O(N^2)$导致EventLoop 事件循环直接卡死
    *   外部微服务发来的高并发请求全部排队堆积。
    *   几百毫秒后，网关和业务服务发生大面积的 Read Timeout（读超时）。
    *   连接池被打满，熔断器被触发，整个系统架构因为底层的一块小内存连锁反应而**瞬间雪崩**

##### Listpack

**Redis7.0彻底抛弃 Ziplist，全面替换为 Listpack**
- Listpack 继承了 Ziplist 空间利用率 100% 和 CPU Cache 友好的优点。
- **但是，它删掉了节点的 `prevlen`！**
    ```text
    +-------------------+-----------------------+--------------------------------+
    | encoding          | content               | element-total-len              |
    | (当前节点编码)    | (真实的业务数据)      | (【当前】节点自身的总长度)     |
    +-------------------+-----------------------+--------------------------------+
    ```
*   **Listpack 的做法是**：“我只记录**我自己**有多长”，并把这个长度信息放在当前节点的**最尾部**。

*  **它是如何实现逆向遍历的？**
   *  当指针需要往前跳时，它直接读取**前一个节点的尾部**（也就是前一个节点的 `element-total-len`），知道了前一个节点的长度，指针一减，就能跳到前一个节点的头部！

*  **完美解决连锁更新：**
   *  现在，如果在 Listpack 中间插入一个 300 字节的大节点 `e_new`。
`e_new` 会在它自己的尾部记录“我占了 300 字节”。
*  **它后面的节点 `e1` 完全不需要做任何改动！** 
   *  因为 `e1` 内部不再维护前面节点的长度。
   *  没有状态耦合，就没有多米诺骨牌。Listpack 兵不血刃地将最差时间复杂度从 $O(N^2)$ 拉回了绝对安全的 $O(N)$。

***

##### Hashtable 的核心结构

**何时必须放弃连续内存？**
- Listpack/Ziplist 是一块紧凑的连续物理内存，它将内存压缩做到了极致。但架构世界永远遵循一个铁律：**时间与空间不可兼得**。
*   **Ziplist/Listpack 是典型的“以时间换空间”**：牺牲了 CPU 的计算时间（必须进行 $O(N)$ 的线性遍历），换取了极致的内存节省。**当 Hash 里的元素只有 50 个时**，**得益于 CPU Cache 的空间局部性**，**线性遍历的速度甚至比算哈希值还快**。
*   **架构的瓶颈**：如果微服务向这个 Hash 中塞入了 10,000 个商品（例如大型 B2B 购物车），线性遍历的 $O(N)$ 就会变成拖垮 Redis 单线程的梦魇。
*   **跃迁触发**：此时，Redis 必须进行底层数据结构的跃迁——抛弃连续内存，重新拥抱离散的指针，回归到**以空间换时间**的经典哈希表模式，以换取绝对的 $O(1)$ 寻址速度。


---

Redis的Hash是嵌套的结构体
- 全局哈希字典也是一个Hash结构，其ht[0]的table[i]的dictEntry链表中存储了目标对象的`redisObject`对象头
*   **最内层：`dictEntry` 既是键值对节点，也是解决哈希冲突的单链表节点**
    *   `*key`：指向 Hash 的字段名（通常是一个 SDS 字符串）。
    *   `v`：既可以存一个指针，指向对象，也可以直接存一个 `uint64_t` 的整数
    *   `*next`：如果发生哈希冲突，则指向下一个 `dictEntry`，通过 **链地址法（单向链表）** 解决哈希冲突。

*   **中间层：`dictht` 底层哈希数组，通过位运算寻址**
    *   它维护了一个**指针数组** `table`。数组的每一个槽位（Bucket）要么是 `NULL`，要么指向 `dictEntry` 链表的头节点。
    *   `sizemask` 永远等于 `size - 1`。在寻址时，Redis 使用高效的位运算 `hash & sizemask` 来代替低效的取模 `%` 运算（与 Java `HashMap` 的底层优化完全一致）。

*   **最外层：`dict`：提供`Hash`的统一API，负责渐进式Rehash**
    * 所有dict**内部都强制绑定了两个哈希表 `ht[0]` 和 `ht[1]`！** 
    * 由于 Redis 是单线程处理命令，如果发生大规模扩容，一次性搬运数据会卡死主线程。
    * `dict` 引入双表机制配合 `rehashidx` 游标，实现了“边运行边搬家”的**渐进式 Rehash**（将在节点 5 深度剖析）。

    ```text
    【字典控制块 dict (万能哈希模板)】
    +---------------------------------------------------------------+
    | type       (类型特定函数指针，用于多态)                       |
    | privdata   (私有数据指针)                                     |
    |                                                               |
    | 【哈希表数组 dictht ht[2]】 (肚子里永远装着两张表)            |
    |  ht[0]:  (日常工作使用的主表)                                 |
    |  +-------------------------------------------------------+    |
    |  | table    (指针数组首地址) ---> [ * ] [ * ] [ * ] [ * ]|    |
    |  | size     (哈希表大小，如 4)     |               |     |    |
    |  | sizemask (掩码，size-1，如 3)   V               V     |    |
    |  | used     (已有节点数量)     [dictEntry]     [dictEntry]    |
    |  +-------------------------------------------------------+    |
    |                                                               |
    |  ht[1]:  (备胎哈希表，平时为空，仅在 Rehash 扩容时启用)       |
    |  +-------------------------------------------------------+    |
    |  | table    ---> NULL                                    |    |
    |  | size     ---> 0                                       |    |
    |  +-------------------------------------------------------+    |
    |                                                               |
    | rehashidx  (渐进式 Rehash 游标，平时为 -1，扩容时记录进度)    |
    +---------------------------------------------------------------+
    ```

- 指针跳转
    ```text
    dict (字典控制块)
    └──> ht[0] (当前工作哈希表)
        └──> table (指针数组)
                └──> table[index] (定位到具体哈希桶)
                    └──> dictEntry (链表头节点) 
                            ├──> key (比对字符串)
                            ├──> v   (获取目标值)
                            └──> *next -> dictEntry (如果哈希冲突，顺着链表往下找)
    ```



---

**哈希冲突采用链地址法和头插法**，**为什么Redis采用头插法？为什么Java废弃头插法改用尾插法？**
  1.  **单线程的绝对优势（无并发死链）**：
      *   Java 1.7 废弃头插法，是因为在**多线程并发扩容**时，头插法会改变链表的节点顺序，极易导致指针成环，触发 `get()` 时的死循环（CPU 100% 灾难）。
      *   而 Redis 的核心工作命令是**绝对单线程**执行的！根本不存在多线程并发操作同一个 dict 的情况，头插法在 Redis 里是绝对安全的。
  2.  **时间局部性原理**：
      *   这是计算机科学中的一个经典原理：**“刚刚被写入的数据，大概率在极短的时间内会被再次访问。”**
      *   使用头插法，最新插入的数据永远在链表的第一个。当微服务马上来 `HGET` 读取刚才写入的数据时，Redis 在槽位里看一眼第一个节点就命中了，连顺着链表往下找的动作都省了，性能榨取到了极致。

- 请注意，**无论头插还是尾插，其时间复杂度都是$O(N)$：为了避免元素重复，有哈希冲突就要检查元素存不存在**

##### hash的编码转换与缓冲命中


Redis 的 Hash 结构为了在省内存和寻址速度之间取得动态平衡，为从连续内存Listpack到离散内存Hashtable的跃迁，划定了两根严格的红线

1.  **`hash-max-listpack-entries`（默认值：512）**
    *   **工作机理**：限制 Hash 中存储的元素（键值对）**总数量**。当向 Hash 中写入新字段时，如果总数量突破了 512 个，底层结构将立刻触发重构，转换为 `dict`。
2.  **`hash-max-listpack-value`（默认值：64 字节）**
    *   **工作机机**：限制 Hash 中任意一个字段（Key 或 Value）的**单体最大字节数**。只要有哪怕一个超级大的字符串（大于 64 字节）被塞入 Hash，即便当前总元素只有 2 个，底层也会瞬间转换为 `dict`。

> 编码转换是**不可逆**的。
> 一旦 Hash 因为某个大 Value 触发了跃迁变成了 Hashtable，哪怕你后来用 `HDEL` 把这个大 Value 删掉了，甚至把元素删得只剩 1 个，它的底层依然是满地指针的 Hashtable，**永远不会**再降级回 Listpack！

**为什么要限制listpack的单体最大字节数？**
- **在连续内存中，跳过一个极其巨大的节点，确实只需要一次 $O(1)$ 的指针加法运算。** 但是单体字节数过大会引发以下灾难
1. 已存在巨大字符串时，节点更新的**拷贝**灾难
   - 如果Listpack有一个巨大的字符串，比如1MB
   - 在该键值对之前的节点有任何变动，都会拷贝，每次都要平移该巨大字符串
2. 插入巨大字符串时的成本
   - 每插入一个巨大字符串，都需要**申请一块新的、更大的连续空间**
   - 离散内存的申请是容易的，绝对连续的内存空间的申请是困难的
3. **CPU 缓存污染**
   - 为了寻找指定的键值对，会执行O(N)时间复杂度的线性查找
   - 为了将巨大的字符串引入高速缓存，会将其他有用的热点数据挤出高速缓存，导致缓存命中率下降


---

调整`hash-max-listpack-entries`、`hash-max-listpack-value`参数的值，本质上是在平衡**内存容量** 与 **CPU 缓存命中率**
- 现代 CPU性能瓶颈在主存，为了弥合速度差，CPU 内部有 L1/L2/L3 高速缓存；CPU 读取内存是按 **Cache Line缓存行64 字节** 整块加载的
- **Listpack 的物理优势**
  - 因为它是一块完全连续的内存，当Listpack元素又少又小时，极大可能会将整个Listpack的大部分节点顺道拉进CPU L1 Cache 中
  - 虽然Listpack 的时间复杂度是 $O(N)$，但是通过预读取，消除了读取内存的瓶颈，**连续内存的 $O(N)$ 甚至比跳跃寻址的 $O(1)$ 还要快**！
- **Hashtable 的指针跳转与缓存未命中**
  - Hashtable时间复杂度理论上变成了完美的 $O(1)$，但是因为Hashtable的节点是**极度离散、随机分布**的，缓冲命中率会非常低
  - 每一次指针的解引用跳转，CPU 的 Cache 大概率会扑空。CPU 只能被迫停下手头的计算，去缓慢的主存里捞数据。一次 Cache Miss，几百个 CPU 时钟周期就白白浪费了。

**调优策略**
- **压榨内存**：在海量冷数据存储场景，并发量不高，将 `entries` 调高到 1024 甚至 2048，省下恐怖的指针内存开销
- **保护Cpu**：在极高频读写的热点路由表场景，把 `entries` 压低到 256，宁愿多花点内存维护指针，也要尽早让底层转为 $O(1)$ 的 Hashtable，从而避免单线程 CPU 因为遍历过长的连续内存而被长期占用，引发微服务的高并发延迟毛刺。
