# 需求设计


秒杀是**一场用极小成本换取极大流量的营销活动。** 这决定它有两条不可逾越的红线：
1. **绝对不能超卖** 
2. **绝对不能影响其他业务** 


**场景 1：秒杀读风暴前夕**
* **时间强同步（防篡改）：** 用户的本地时间不可信。系统提供轻量级时间校准接口，强制依赖服务端时间，只有当服务端到达开闸时间，抢购才真正开始。
* **抢购动态 URL（防抓包）：** 彻底阻断黑客写脚本提前请求。倒计时结束瞬间，前端必须先获取动态加密的“抢购令牌（Token/Salt）”并拼接至真实 URL。网关层直接踢掉所有无令牌的越权访问。
* **多品类混合秒杀（热点与资源隔离）：** 奖池分级设置（100个手机、200个台灯、400个牙刷）。在业务上保证海量抢手机的用户不会挤爆系统，从而影响少量抢牙刷用户的体验
* **全局限购一件（跨商品并发对抗）：** 回归“交个朋友”的商业本质，严禁通吃。针对黑客同时发起多品类抢购的并发攻击，系统必须具备跨商品的原子级校验能力，确保用户一旦命中一件商品，其所有的并发请求立刻被拦截阻断。允许用户在抢购失败后，再次抢购或抢购其他商品

---

**场景 2：开闸瞬间**
* **瞬间洪峰与极速失败宣告：** 
  * 业务设定：开闸瞬间 10 万并发请求涌入争抢 100 个真实库存。
  * 能力边界：系统必须在毫秒级完成拦截与抉择。对于未能抢到的 99,900 个请求，立刻明确返回“抢购失败”。严禁让用户处于“转圈加载”的排队等待状态，彻底杜绝无效请求长期占用后端连接和内存资源。
* **宏观热点分治（库存分片 Sharding）：** 
  * 业务设定：放弃单一入口的全局锁，采用“局部公平”换取系统极高的吞吐量。
  * 能力边界：针对单商品极热并发导致的单点物理瓶颈（网卡/CPU），系统需具备分治能力。提前将 100 个库存拆分为 10 个桶（Bucket），预热至 10 个独立缓存分片。流量经网关后通过一致性 Hash 路由均匀打散，将 10 万 QPS 的单点风暴降维为单节点 1 万 QPS 的局部压力。
* **微观原子扣减（防超卖底层红线）：** 
  * 业务设定：在单一库存分片内部，必须严格保证绝对的时序公平与数据一致性。
  * 能力边界：引入 Redis Lua 脚本，将“用户购买资格校验”与“库存扣减”合并为不可分割的原子操作。在最底层彻底斩断高并发下的竞态条件（Race Condition），确保系统在任何极端并发下，绝对不超卖一件商品。

---

**场景 3：订单生命周期与资产回收**

* **限时锁定机制（业务设定）：**
  * 抢购成功仅代表“购买资格锁定”。用户必须在严苛的规定时间（如 5 分钟）内完成最终支付。超时未支付的订单必须被强制取消。
* **资产静默回滚（捡漏业务流转）：**
  * 放弃复杂的顺延或通知机制。超时释放的库存将直接、静默地退回全局可用库存池。此时仍在前台尝试请求抢购接口的用户，有几率直接命中被释放的库存，实现业务上的“捡漏”。
* **事件驱动的延时释放（底层技术边界）：**
  * 严禁使用基于数据库的传统定时轮询（扫表）机制处理超时订单，避免对核心数据库造成毁灭性打击。
  * 系统必须引入基于消息队列的**延时队列（Delay Queue / 死信队列 DLX）**。在下发购买资格的瞬间，投递一条带有 5 分钟 TTL（生存时间）的延时消息。时间到达后，由异步消费者核验支付状态，并在内存与数据库层面执行极低损耗的库存回滚闭环。
---

**场景 4：恶意的商业破坏与反欺诈**
* **网关层精准限流（底层物理防线）：**
  * 针对恶意脚本（如单一 IP 或 UserID 的超频请求），在网关或前置拦截器层接入基于 Redis Lua 实现的**令牌桶/滑动窗口算法**。对极值请求进行硬性物理熔断，彻底阻断无效流量穿透至后端核心交易引擎。
* **基于水位的动态降级（人机验证与业务削峰）：**
  * 摒弃一刀切的风控策略，采用**基于流量水位的动态容灾机制**。流量平稳时，直接下发动态抢购令牌；当瞬时并发触及限流阈值时，网关动态阻断，并下发特定风控状态码（如 `Code: 4001`），强制触发算术题或滑块验证。
  * 利用人类解答验证码的固有耗时差异（1~3秒不等），强行将 $T=0$ 时刻瞬间爆发的极高并发请求，在时间轴上平滑打散，极其优雅地完成系统的第一波**业务级削峰**。
* **状态码驱动的无缝协同（极致用户体验与防刷闭环）：**
  * 严禁“多次点击”的割裂交互。前端网络拦截器在捕获风控状态码后，自动缓存用户的抢购上下文并唤起验证组件。用户解答完毕后，前端底层代码静默携带验证凭证完成 URL 换取与最终的订单冲锋。
  * 整体链路在用户视角仅为“单次点击触发”，在无情绞杀自动化脚本的同时，最大程度保障了真实用户的丝滑交易体验。

# 怎么根据需求文档设计系统？

大哥非常欣赏你这个决定！这才是做高阶项目的正确姿态。

如果你只是拿着我画好的架构图去背，面试官只要稍微改变一个条件（比如问：“如果不用 Redis，只用 MySQL 怎么抗？”），你马上就会原形毕露。但如果你是**自己推演**出来的，哪怕你的技术栈选型很稚嫩，你也能讲清楚**“我为什么这么选”**，这才是架构师的灵魂。

既然你缺乏架构经验，那大哥今天不给你鱼，直接给你一把**“架构师的屠龙刀”**。

不管是做秒杀、做电商还是做外卖，所有从“业务需求”推演到“技术架构”的过程，都逃不出以下这 **“四步推演法则”**。你现在就拿着我们刚刚确定的四大业务场景，跟着我这四步来走：

---

### 第一步：画出最原始的“单机裸奔”数据流（生命周期法）
忘掉那些高大上的微服务、Redis、MQ。假设你现在只有一台破电脑，一个 Tomcat 跑着 Java 代码，连着一个 MySQL。
你首先要在脑海里（或纸上）画出一个用户的请求，从他的手机，最终落到数据库的**一条线**：

> **手机 App/浏览器** $\rightarrow$ **网络** $\rightarrow$ **你的 Java 服务器（Tomcat）** $\rightarrow$ **数据库（MySQL）**

这就是系统的“骨架”。所有的复杂架构，都是在这个骨架上长出肉来的。

### 第二步：把业务场景当成“炸弹”，往骨架上扔（寻找瓶颈）
现在，把你写好的四大业务场景，一个个扔向上面那个脆弱的“单机裸奔”骨架，看它会在哪里被炸毁：

* **扔场景 1（读风暴与防抓包）：** 100万人疯狂刷新页面。
    * *炸弹落点：* 你的 Java 服务器和 MySQL 会瞬间被这 100 万个查询请求压垮。
    * *你需要寻找的武器：* 我怎么才能让这 100 万个请求**“不碰到”**我的 MySQL？甚至**“不碰到”**我的 Java 服务器？
* **扔场景 4（黄牛并发与风控）：** 机器脚本瞬间发起一万次点击。
    * *炸弹落点：* 所有的恶意请求都进入了 Java 代码，开始消耗 CPU。
    * *你需要寻找的武器：* 我怎么能在 Java 业务代码执行之前，就建一堵墙，把坏人挡在外面？
* **扔场景 2（写冲突与分片）：** 10 万人瞬间要求扣减同一个数据库字段。
    * *炸弹落点：* MySQL 的行级锁被严重冲突，发生死锁或超时。
    * *你需要寻找的武器：* 我怎么才能不在 MySQL 里做减法？去哪里做减法最快？
* **扔场景 3（超时释放）：** 订单生成后要等 5 分钟，不行就回滚。
    * *炸弹落点：* 如果用 Java 线程死等 5 分钟，服务器内存马上撑爆。
    * *你需要寻找的武器：* 我需要一个什么“备忘录”，能帮我记住这件事，并且在 5 分钟后准时叫醒我？

### 第三步：对症下药，引入架构组件（技术选型）
基于你在第二步找到的“痛点”，现在开始像搭积木一样，在骨架上插入对应的技术组件。这是你需要自己去查阅资料和填空的地方。

* **为了挡住读风暴（场景1）：** 你会发现在“手机”和“Java 服务器”之间，需要插入 **CDN（内容分发网络）** 和 **Nginx（反向代理）**；在“Java 服务器”和“MySQL”之间，需要插入 **Redis（缓存）**。
* **为了挡住黄牛（场景4）：** 你会发现在进入业务代码之前，需要一个**网关层（API Gateway）**来做限流和风控。
* **为了解决极速扣库存（场景2）：** 你决定把原本在 MySQL 里的操作，转移到 **Redis（配合 Lua 脚本）** 中去完成。
* **为了解决延时任务（场景3）：** 你发现在 Java 服务器的旁边，需要挂载一个 **MQ（消息队列）**。

### 第四步：连点成线，梳理高可用防线（分层架构图）
当你把所有组件选好后，你需要把它们按照“离用户的距离”进行分层排布。一个标准的后端架构图，一定是自上而下、由外及内的。

你可以按照这个经典的**“四层漏斗模型”**来构思你的架构草图：

1.  **端到端防线（Client & Edge）：** 页面长什么样？CDN 怎么缓存静态资源？
2.  **接入与风控层（Gateway）：** 怎么做动态路由？怎么限流？风控验证码在哪触发？
3.  **核心业务层（App & Cache）：** 你的微服务/单体应用长什么样？Redis 怎么做预热和库存分片？Lua 脚本在哪执行？
4.  **异步与持久化层（Async & DB）：** 抢购成功后，怎么通过 MQ 异步写数据库？超时怎么回滚？
