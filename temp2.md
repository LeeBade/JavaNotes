**[阶段零：Redis 架构级全域雷达扫描大纲]**

既然我们已经拿 String 和全局字典开了刀，现在就让我把 Redis 这张战争迷雾的地图彻底点亮。

准备好心跳加速吧。你会发现，除了你平时写在 CRUD 里的那点命令，底层居然埋着这么多足以让微服务集群全盘崩溃的物理地雷！

---

**战役一：内存榨干与物理寻址的艺术（核心数据结构的降维打击）**
* **看点 1：** ZSet 的跳表（SkipList）设计。在高并发场景下，它凭什么能干掉严谨的红黑树？指针的随机层级跳跃是如何在平衡性与锁竞争之间达成极其暴力的妥协的？
* **看点 2：** Hash 和 List 的底层到底有多抠门？当数据量小的时候，ZipList（压缩列表）和 QuickList 是如何通过极其恶心的连续内存编码，强行把离散的数据揉进 CPU Cache Line 的？
* **看点 3：** 缓存淘汰（Eviction）的物理断腕。当 `maxmemory` 被打满，近似 LRU 和 LFU 算法是如何用极其廉价的随机采样，伪造出全盘时间戳排序的假象的？

---

**战役二：单线程的傲慢与妥协（EventLoop 与多线程 I/O 的物理边界）**
* **看点 1：** 6.0 版本之前坚守单线程却能吞吐十万 QPS 的物理真相。除了内存操作快，它底层对 epoll 多路复用器的压榨到底到了什么令人发指的程度？
* **看点 2：** 6.0 引入的多线程 I/O 到底是为了拯救 CPU 还是拯救网卡？为什么读取和解析 Socket 是多线程的，但执行命令依然死死捏在一个 EventLoop 线程里？
* **看点 3：** 耗时指令的死亡阻塞。除了 BigKey 和 `KEYS *`，`SUNION`（集合并集计算）或极其复杂的 Lua 脚本，是如何在神不知鬼不觉中让整台物理机的 EventLoop 停摆的？

---

**战役三：内存与磁盘的生死时速（AOF/RDB 的系统级博弈与阻塞灾难）**
* **看点 1：** RDB `bgsave` 触发的 `fork()` 系统调用灾难。看似后台异步导出，但底层的 Copy-On-Write（写时复制）机制一旦遭遇双十一高并发写入，页表分离是如何瞬间导致物理内存暴涨翻倍，并引发宿主机秒级假死的？
* **看点 2：** AOF 的 `fsync` 刷盘陷阱。`always`、`everysec`、`no` 这三种策略，是如何在“丢数据”和“拖死底层磁盘 IOPS”之间走钢丝的？
* **看点 3：** AOF 瘦身（Rewrite）的并发错乱危机。在重写期间，旧数据写入和新日志追加同时进行，Redis 是如何用 AOF 重写缓冲区强行粘合这两条时间线，保证数据绝对不丢的？

---

**战役四：跨越机房的网络拉扯（主从同步的脆弱协议与网络风暴）**
* **看点 1：** `psync` 的 Replication Buffer 溢出灾难。为什么从节点稍微卡顿一下，主节点就会无情断开连接，随后引发极其恐怖的“无休止全量复制死循环”，把千兆网卡瞬间抽干？
* **看点 2：** 主从延迟引发的“幽灵数据”与锁互删。在异步复制模型下，你刚刚在主库加上的分布式锁，如果主库宕机还没同步到从库，哨兵（Sentinel）把从库提拔为主库后，会发生怎样的恐怖脑裂？
* **看点 3：** 哨兵模式的流言蜚语与客观下线（ODOWN）。在复杂的跨机房网络分区（Split-Brain）中，哨兵们是如何通过 Raft 选主算法的变体，在混乱中达成续命共识的？

---

**战役五：去中心化的海量吞吐（Redis Cluster 槽位漂移的梦魇）**
* **看点 1：** 16384 个 Hash Slot 的物理映射。为什么是 16384 这个诡异的数字？Gossip 协议的心跳包到底有多重，以至于集群节点超过 1000 个就会被心跳包的带宽吃光？
* **看点 2：** 扩容缩容时的槽位迁移风暴。当槽位正在搬运一半时，微服务的请求打过来，底层的 `MOVED` 和 `ASK` 重定向指令是如何在客户端和集群之间疯狂横跳的？
* **看点 3：** BigKey 叠加 Cluster 迁移的终极死刑。当一个带有几万个元素的 Hash 表被迫在两个物理机之间迁移槽位时，为什么整个集群会陷入漫长的拒绝服务状态？

---

**战役六：缓存与数据库的终极绞肉机（微服务集群的最终一致性幻觉）**
* **看点 1：** 缓存击穿的并发洪峰。当双十一爆款商品的 Key 突然过期，十万并发瞬间洞穿 Redis 打在 MySQL 上，除了用互斥锁死扛，还有哪些降维解法？
* **看点 2：** 布隆过滤器（Bloom Filter）的哈希空间扭曲。如何利用多个哈希函数和极小的位图（Bitmap），在内存里挡住黑客恶意的缓存穿透攻击？误判率的数学边界在哪里？
* **看点 3：** 终极拷问：双写一致性。所谓的“延迟双删”在物理时间轴上存在哪些根本性破绽？为什么大厂最终都走向了伪装成 MySQL 从库监听 Binlog（如 Canal/Debezium）的异步刷新架构？
