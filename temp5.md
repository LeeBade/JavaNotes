你的疑惑非常精准！InnoDB 的锁机制，尤其是行级锁的具体应用，确实是 MySQL 中最复杂、最容易让人绕进死胡同的部分。很多开发者在这里都会感到困惑。

我们把你的两个问题掰开揉碎了来看。

### 问题一：行锁和表锁要么是排他锁（X）要么是共享锁（S）对吗？

**基本是对的，但我们要把“锁的模式”和“锁的粒度”区分开来理解。**

* **排他 (X) 和共享 (S)** 指的是锁的**基本模式（读写属性）**。
* **表锁和行锁** 指的是锁的**粒度（锁定范围）**。

它们是可以自由组合的：
1.  **表级锁** 可以是 S 锁（表共享读锁），也可以是 X 锁（表独占写锁）。上一篇提到的**意向锁 (IS/IX)** 也是表级锁的一种特殊模式，它本质上也是基于 S 和 X 衍生出来的。
2.  **行级锁** 同样有 S 锁（比如执行 `SELECT ... FOR SHARE`）和 X 锁（比如执行 `UPDATE` 或 `SELECT ... FOR UPDATE`）。

所以，无论锁的范围多大，它最终都要落实到“允不允许别人读”和“允不允许别人写”这两个根本属性上。

### 问题二：在非快照读下，记录锁、间隙锁、临键锁的应用场景到底是什么？

首先，我们要**纠正一个非常普遍的误区**：你提到“只要不是快照读（当前读），都应该使用了行锁对吧”。**这句话不完全对！**

**核心原则：InnoDB 的行锁是加在「索引」上的，而不是加在真实的数据行上的。**
如果你的 SQL 语句（比如 `UPDATE`）**没有用到索引**，或者 MySQL 优化器认为全表扫描比走索引快，那么 InnoDB 会对表中的所有记录和间隙加锁，这在并发表现上**等效于锁死了整张表**！这是导致线上生产环境死锁或性能骤降的常见元凶。

明确了“行锁依赖索引”之后，我们来看看具体那三种锁的应用场景。

> **大前提：** 以下讨论均基于 InnoDB 的默认事务隔离级别 —— **可重复读 (Repeatable Read, RR)**。在“读已提交 (RC)”级别下，基本只有记录锁，没有间隙锁（除外键检查等极少情况）。



#### 1. 记录锁 (Record Lock)
* **本质：** 锁住具体的某一条（或多条）索引记录。仅仅是记录本身，不涉及记录之间的空隙。
* **核心应用场景：** **精准匹配，且具有唯一性。**
* **具体触发条件：** 使用**主键**或**唯一索引**进行**等值查询**，且命中记录。
* **举个例子：**
    ```sql
    -- id 是主键，且 id=5 的记录存在
    UPDATE user SET age = 20 WHERE id = 5;
    ```
    此时，InnoDB 只会给 `id=5` 的这一行主键索引加上记录锁（X锁）。其他事务可以自由修改 `id=4` 或 `id=6` 的数据。

#### 2. 间隙锁 (Gap Lock)
* **本质：** 锁住两个索引记录之间的“空隙”，**不包括任何记录本身**。
* **核心目的：** 唯一的目的就是**防止其他事务在这个空隙中执行 `INSERT` 操作**，从而解决“幻读”问题。
* **核心应用场景：** **范围查询，或者普通索引的等值查询。**
* **具体触发条件：**
    * 使用非唯一索引进行等值查询。
    * 使用主键/唯一索引进行范围查询。
    * 等值查询一条不存在的记录。
* **举个例子：** 假设表中有 `id` (主键): 1, 5, 10。
    ```sql
    -- 查找一个不存在的记录
    SELECT * FROM user WHERE id = 7 FOR UPDATE;
    ```
    因为 `id=7` 不存在，InnoDB 会在 `id=5` 和 `id=10` 之间加上**间隙锁**。此时，如果别的事务想插入 `id=6`、`id=8`，就会被阻塞。但其他事务仍然可以修改 `id=5` 或 `id=10`（因为只锁了间隙）。

#### 3. 临键锁 (Next-Key Lock)
* **本质：** **记录锁 + 间隙锁**。它不仅锁住一个区间，还会锁住这个区间的“右边界”记录（左开右闭区间，例如 `(5, 10]`）。
* **核心应用场景：** **这是 InnoDB 在 RR 隔离级别下的默认行锁算法！**
* **运作机制（动态退化）：**
    其实你不需要刻意去记何时触发临键锁，你只需要知道，InnoDB 默认上来就是加临键锁，然后根据索引的类型进行**退化**：
    1.  如果遇到**主键/唯一索引的等值查询**，且记录存在，临键锁会**退化为“记录锁”**（因为唯一性保证了不需要锁间隙来防幻读）。
    2.  如果遇到**等值查询但记录不存在**，临键锁会向下**退化为“间隙锁”**。
    3.  如果是**范围查询**，或者使用**非唯一普通索引**，它就会老老实实保持 **临键锁** 的形态，既锁记录，又锁间隙。

### 总结：如何判断用哪种行锁？

你可以用这个逻辑链条来判断一个当前读（`UPDATE`, `DELETE`, `FOR UPDATE`）到底加了什么锁：

1.  **有没有走索引？** 没走索引 -> 全表记录+间隙锁定（类似表锁）。走了索引 -> 往下看。
2.  **走的是唯一索引/主键，还是普通索引？**
    * **唯一/主键 + 等值匹配 + 存在该记录：** 退化为 **记录锁**。
    * **普通索引 + 等值匹配：** **临键锁**（锁住记录和前面的间隙）+ **间隙锁**（锁住记录后面的间隙）。
3.  **是不是范围查询（`>`, `<`）？** 保持 **临键锁**，锁住扫描到的所有区间和边界记录。
