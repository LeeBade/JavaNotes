要准确回答这两条 SQL 语句的加锁情况，我们需要以 MySQL 的 **InnoDB 存储引擎**为例，并假设当前的事务隔离级别是默认的**可重复读（Repeatable Read, RR）**。

在 InnoDB 中，加锁的行为与 `SCORE` 字段**是否有索引**以及**索引的类型**息息相关。我们分情况来详细拆解：

---

### 第一部分：执行 `SELECT * FROM TEST WHERE SCORE=90`

这是一条普通的查询语句，在 InnoDB 中被称为**快照读（Snapshot Read）**。

* **加锁情况**：**不加任何锁**。
* **原理**：InnoDB 利用 MVCC（多版本并发控制）机制，读取的是该记录的一个历史快照版本。即使其他事务正在修改 `SCORE=90` 的数据，这条语句也能正常读取，不会被阻塞，也不会去阻塞别人。

---

### 第二部分：执行 `SELECT * FROM TEST WHERE SCORE=90 FOR UPDATE`

这是一条**当前读（Current Read）**语句，加上了 `FOR UPDATE`，意味着你要对读到的数据加**排他锁（X锁）**。

具体的加锁方式取决于 `SCORE` 字段的索引情况。为了方便理解，假设表中目前存在三条记录，对应的 `SCORE` 分别是 `80`、`90`、`100`。

#### 场景 1：`SCORE` 是唯一索引（或主键）
因为是唯一索引，数据库知道 `SCORE=90` 的记录最多只有一条。

* **如果存在 `SCORE=90` 的记录**：
    * **加锁**：只加**记录锁（Record Lock）**。
    * **作用**：仅锁住 `SCORE=90` 这一行索引节点，允许其他事务在 90 之前或之后的间隙插入新数据。
* **如果不存在 `SCORE=90` 的记录**（比如你查的是 85）：
    * **加锁**：Next-Key Lock 会退化为**间隙锁（Gap Lock）**。
    * **作用**：锁住 `(80, 90)` 这个间隙，防止其他事务在这个区间内插入数据（防止幻读）。

#### 场景 2：`SCORE` 是普通非唯一索引（最常见考点）
因为是非唯一索引，数据库无法确定 `SCORE=90` 的记录有几条，为了防止**幻读**（即其他事务又插入了一条 `SCORE=90` 的记录），它必须把周边的间隙也锁住。

* **加锁**：会同时加上**临键锁（Next-Key Lock）**、**间隙锁（Gap Lock）**和**记录锁（Record Lock）**。
    1.  **Next-Key Lock**：在 `SCORE` 的辅助索引上，锁住前一个区间及当前记录，即加锁区间为 `(80, 90]`。
    2.  **Gap Lock**：为了防止其他事务在此记录之后插入 `SCORE=90`，会在下一个区间加上间隙锁，即加锁区间为 `(90, 100)`。
    3.  **Record Lock**：在聚簇（主键）索引上，对所有满足 `SCORE=90` 的主键数据行加上记录锁（防止别人 UPDATE 或 DELETE 真实的数据行）。
* **总结**：在辅助索引上，实际上锁住的是 `(80, 100)` 这个开区间；在主键索引上，锁住的是具体的行。

#### 场景 3：`SCORE` 字段没有索引
如果 `SCORE` 没有任何索引，InnoDB 无法通过索引树去精准定位数据，只能进行**全表扫描**。

* **加锁**：会给表中的**每一条记录**加上 Next-Key Lock，也就是锁住了所有的记录以及所有的间隙。
* **后果**：这等同于把整张表都锁死了（类似表锁），其他事务将无法插入、删除或更新任何记录，直到当前事务提交。这在生产环境中会导致严重的性能瓶颈。

---

### 总结对照表

| SQL 语句 | 索引情况 | 加锁类型及范围 (假设 RR 隔离级别) |
| :--- | :--- | :--- |
| `SELECT ... SCORE=90` | 任何情况 | **不加锁** (MVCC 快照读) |
| `SELECT ... FOR UPDATE` | **主键/唯一索引** (记录存在) | **记录锁** (仅锁所在行) |
| `SELECT ... FOR UPDATE` | **普通非唯一索引** | **临键锁 + 间隙锁** (锁住前后间隙) + 主键**记录锁** |
| `SELECT ... FOR UPDATE` | **无索引** | **全表扫描，锁住所有记录及所有间隙** (极度危险) |
