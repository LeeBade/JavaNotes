#### 内存优化：TransientStorePool与PageCache调优
##### Page Cache的锁争用与写毛刺根因

**具体的工程问题**

* **工程问题一：OS内核刷盘动作引发的“写阻塞反噬”（毛刺灾难）**
  * Mmap 的核心逻辑是将写入动作交给了 OS 的 Page Cache，业务以为自己写的是纯内存。但 Page Cache 归根结底是要落盘的。
  * 当积累的脏页过多，操作系统内核的后台线程介入并强制将内存数据刷入缓慢的物理磁盘时。
  * 为了保证数据在传输过程中的绝对一致性，内核会对这块内存页加锁。
  * 如果此时业务线程恰好通过 Mmap **向同一块内存页发起持续的极速追加**，**业务线程将被内核底层的读写锁无情挂起**，写入延迟从微秒级瞬间暴增至数百毫秒甚至几秒，这就是监控大盘上令人胆战心惊的“写毛刺”。


* **工程问题二：读写同源导致的高并发资源互斥（读写锁争用）**
  * RocketMQ 默认架构下，生产者追加消息（写 CommitLog）和消费者拉取消息（读 CommitLog）操作的物理载体，都是同一块 Page Cache 空间。
  * 在遇到大量冷数据回溯消费时，消费者会疯狂触发磁盘的随机读，将大量冷数据加载到 Page Cache 中，不仅会把原本写入的热数据挤出内存，更会在操作系统底层频繁触发内存页的换入换出锁，导致主写链路的 TPS 遭受无妄之灾。



---

**技术执行流**

* 为了理解毛刺的产生，我们必须下沉到 Linux 内核态，推演 Mmap 在遭遇高强度脏页积压时，是如何从“极速的内存赋值”退化为“阻塞的磁盘 I/O”的。
* **引入静态组件**：
  * **业务写线程（Producer Thread）**：负责通过 `MappedByteBuffer.put()` 向内存写数据。
  * **Page Cache（内核页缓存）**：Linux 管理的文件系统缓存，分为 Clean Page（干净页）和 Dirty Page（脏页）。
  * **pdflush/flusher（内核刷盘线程）**：Linux OS 内部的守护线程，负责将 Dirty Page 写回磁盘。
  * **Page Lock / i_mmap_rwsem（内核级内存锁）**：Linux 内核用来保护内存页状态、防止并发读写引发数据错乱的底层互斥锁。


* 极高并发下触发内核写阻塞的灾难执行流：
  1. **常态极速追加**：
     * 业务写线程通过 Mmap 不断向虚拟地址空间赋值，OS 在后台悄悄分配 Page Cache 物理页。
     * 此时内存充足，写入极快（纳秒/微秒级），产生大量脏页。


  2. **触碰内核警戒水位**：
     * 当内存中的脏页总比例达到 Linux 内核设定的阈值（如 `vm.dirty_background_ratio`，默认 10%）时。


  3. **内核线程觉醒与 DMA 锁定**：
     * Linux 内核的 **flusher 线程** 被强制唤醒，开始批量将这部分脏页通过 DMA（直接内存访问）控制器搬运到磁盘控制器。
     * 为了防止搬运到一半时内存数据被用户态应用修改，内核会对这批正在刷盘的物理页加上 **Page Lock**。


  4. **业务线程撞锁（毛刺发生）**：
     * 就在这几毫秒内，高并发的**业务写线程**恰好执行到了同一个物理页的追加写入位置。
     * 业务线程在用户态调用 `put()`，触发内核的寻址，发现目标页被锁。业务线程被迫发生上下文切换，从运行态跌入**深度休眠状态（D状态：Uninterruptible Sleep）**。


  5. **物理 I/O 决定耗时**：
     * 业务线程此时的挂起时间，完全取决于那块缓慢的机械硬盘完成这次 I/O 的时间。


  6. **锁释放与延迟突刺**：
     * 磁盘 I/O 结束，内核解开 Page Lock，唤醒业务线程。
     * 业务代码中的一次简单内存赋值，硬生生耗费了 200 毫秒。对于 RocketMQ 而言，这就意味着这一瞬间的并发吞吐量归零。





---

**底层原理**

1. **DMA 一致性与内存锁定**
   * 计算机体系结构决定了，当 DMA 正在把主存（RAM）的数据送到外设（Disk）时，必须保证这块主存的内容绝对静止。
   * 如果允许 CPU 继续修改这块内存，磁盘上就会写下撕裂的半成废品。因此，OS 级别对刷盘页的物理锁定是不可逾越的硬件底线。


2. **Page Cache 的“全局公地”悲剧**
   * Mmap 仅仅是映射，它并没有真正“独占”这块内存。Page Cache 是操作系统全局的共享资源，它受到整台物理机上其他进程（如日志采集器、其他中间件）内存抢占的影响。这种资源调度上的“非决定性（Non-deterministic）”，是 Java 应用层极其难以掌控的。



---

**架构决策与API映射**

* **对应工程问题一与二：快速失败（Fast-Fail）的防御机制**
  * **决策思路**：
    * 在极高并发的集群中，单台 Broker 发生短暂的写毛刺是物理宿命。
    * 如果业务线程死死排队等待这台 Broker 的 OS 锁释放，会导致整个微服务链路级联超时。
    * 正确的做法是：如果 Broker 发现自己正在遭遇 OS 毛刺，应该**立刻拒绝服务**，迫使 Producer 快速收到失败响应，并利用重试机制将流量路由到集群中其他健康的 Broker 上。


    ```java
    // 【核心配置】OS Page Cache 繁忙超时时间
    // 如果 Broker 内部抢占锁的时间超过此时长（默认 1000 毫秒），直接丢弃该写入请求，返回 SYSTEM_BUSY。
    osPageCacheBusyTimeOutMills = 1000 

    ```



---

**边界与局限**

* **无法被应用层彻底根除**：
  * 只要你的写入模型依然是 `业务线程 -> OS Page Cache -> 磁盘`，且读写都在同一片 Page Cache 中混战。
  * 应用层无论写多么优雅的并发代码，都无法从根本上消除 Linux 内核在进行物理内存换页和刷盘时带来的微小停顿。这促使架构师必须去寻找“跳出三界外，不在五行中”的绝对纯粹的堆外内存隔离方案。



---

**关键点**

* Mmap 并非毫无代价，它的阿喀琉斯之踵在于：当 OS 后台刷盘时，会触发内核级的物理内存锁（Page Lock）。
* **业务写线程一旦撞上正在被 OS 刷盘保护的内存页，就会从极速的内存访问退化为同步的磁盘 I/O 等待，形成严重的写毛刺**。
* **读写同源（消费者回溯读引发的 Page Cache 淘汰）会进一步加剧内核级的内存锁争用**。

> 架构直觉：借用操作系统的缓存（Page Cache）就像是向高利贷借钱。平常资金周转极快（吞吐量高），但当债主（OS内核）突然上门收账（强制刷盘与回收内存）时，你的业务线程只能乖乖停下手中所有的活儿，接受物理层面的绝对制裁。要想摆脱制裁，就必须自己建一个不受 OS 管辖的“私人金库”。

##### TransientStorePool堆外内存池机制

**具体的工程问题**

* **工程问题一：如何在应用层开辟一块绝对免疫“OS刷盘锁”的法外之地？**
  * Node 1 揭示了毛刺的根因：只要写操作直接触碰 Page Cache，就必然存在与 OS 刷盘线程撞锁的风险。
  * 要想彻底根除这种由于内核机制带来的不可控毛刺，业务写入的第一落点必须**完全脱离操作系统的管辖**。
  * 我们**需要在 JVM 进程的用户态空间内，寻找一块完全属于自己的、绝对不会被 OS 后台偷偷锁住的纯粹内存。**


* **工程问题二：如何避免堆内缓存引发的 JVM 垃圾回收（GC）风暴？**
  * 如果这块“私人金库”直接开辟在 JVM 的 Heap（堆）内存中（比如 `byte[]` 数组），当面对 1GB 甚至几 GB 的海量消息写入时，会迅速占满老年代，引发极其致命的 Full GC（Stop The World），导致应用完全僵死。
  * 我们需要一种**既在用户态、又不受 JVM GC 管控的特殊内存**。



---

**技术执行流**

* 为了获得绝对纯粹的物理隔离并规避 GC，RocketMQ 引入了基于 `DirectByteBuffer` 的堆外内存池（TransientStorePool）机制，将内存的控制权彻底收归应用层。
* **引入静态组件**：
  * **TransientStorePool（临时存储池）**：RocketMQ 内部管理的堆外内存池对象。
  * **DirectByteBuffer（直接内存）**：Java NIO 提供的脱离 JVM 堆管理的底层原生内存结构（Off-heap Memory），由 C 语言底层的 `malloc` 分配。
  * **mlock（内存锁定操作）**：在此处再次被利用，确保堆外内存不会被 OS 换出到 Swap 区。


* 堆外内存池的初始化与借还流转执行流：
  1. **池化预分配（进程启动阶段）**：
     * Broker 启动时，`TransientStorePool` 会一次性向操作系统申请若干个（默认 5 个）大小严格为 1GB 的 **DirectByteBuffer**。
     * *易错点：这 5GB 内存完全独立于 Page Cache，且不属于 JVM 堆。它是纯粹的用户态物理内存。*


  2. **物理锁定与系统初始化**：
     * 利用 JNI 对这 5 个 1GB 的 DirectByteBuffer 依次执行 `mlock`，将其死死钉在物理内存中。然后将它们放入一个双端队列（Deque）中备用。


  3. **借用（Borrow）以承接极限流量**：
     * 当创建一个新的 1GB CommitLog 文件时，引擎不再直接让业务线程去写 Page Cache（MappedByteBuffer）。
     * 引擎从 `TransientStorePool` 的队列中**借出**一个 1GB 的 DirectByteBuffer，交由这个新建的 MappedFile 独占。


  4. **极速写入（Write）**：
     * 业务写线程到来，将消息完全以纯内存赋值的方式，写入这块借来的 DirectByteBuffer 中。
     * 此时没有任何内核介入，没有任何系统调用，没有任何 OS 刷盘锁的干扰，写入耗时被绝对压平在完美的纳秒/微秒级。


  5. **归还（Return）与循环利用**：
     * **当这块 1GB 的 DirectByteBuffer 中的数据被后台线程完整地“提交（Commit）”到了 Page Cache 后，它会被清空**。
     * 引擎将其**归还**给 `TransientStorePool`，**供下一个新建的 CommitLog 文件继续循环使用。**





---

**底层原理**

1. **用户态内存自治（User-space Memory Autonomy）**
   * 操作系统管理 Page Cache 时，要兼顾全局进程和磁盘硬件的步调，因此充满了各种锁与权衡。
   * `TransientStorePool` 的本质是放弃对内核的依赖，在用户空间内手动实现了一个极其粗暴但专一的“**一级缓存**”。


2. **池化技术（Pooling）与零内存分配开销**
   * 动态申请 1GB 的堆外内存是一项极其昂贵的系统调用。RocketMQ 在启动时一次性吃掉 5GB 内存并池化循环使用，彻底消灭了运行时的内存分配开销和内存碎片问题。



---

**架构决策与API映射**

* **对应工程问题一与二：极致平滑的终极开关**
* **决策思路**：只有在业务对“写毛刺”极其敏感、追求绝对的写入平滑，且机器物理内存极其充裕的情况下，才应当开启此开关。


  ```ini
  # 【核心配置】是否开启堆外内存池（默认 false）
  # 强烈注意：此开关必须配合 flushDiskType = ASYNC_FLUSH（异步刷盘）使用。
  # 开启后，主干写链路将彻底脱离 Page Cache 的束缚。
  transientStorePoolEnable = true 

  # 堆外内存池中 DirectByteBuffer 的个数，默认为 5（即额外占用 5GB 物理内存）
  transientStorePoolSize = 5 

  ```



---

**边界与局限**

* **灾难降级：从“掉电丢数据”恶化为“进程崩溃丢数据”**：
  * 这是引入 `TransientStorePool` 最致命的物理代价。原本数据直接进入 Mmap，只要 JVM 崩溃（如 OOM 被杀、`kill -9`），数据还在内核 Page Cache 里，不会丢失。
  * 但现在第一落点变成了 JVM 用户态的 DirectByteBuffer。**只要 JVM 进程一死，堆外内存立刻被 OS 回收，所有还没来得及提交到 Page Cache 的消息将瞬间灰飞烟灭**。系统的高可用性底线被严重击穿。


* **极其奢侈的内存开销（双倍空间）**：
  * 开启此机制后，一份 1GB 的数据，在物理机上会同时存在两份内存镜像：一份在应用层的 `TransientStorePool` 中，另一份在内核层的 `Page Cache` 中。这要求 Broker 所在物理机必须预留极大的冗余内存空间，否则极易触发 OS 的 OOM-killer。



---

**关键点**

* `TransientStorePool` 通过池化预分配 1GB 规格的 DirectByteBuffer，在 JVM 堆外开辟了极致的写入缓冲。
* 写操作的第一落点变成了用户态物理内存，彻底摆脱了 OS 刷盘机制和 Page Lock 的干扰，压平了所有写毛刺。
* 享受极致写入平滑的代价是极其高昂的：它会让数据的抗灾能力降级（进程崩溃即丢数据），并带来双倍的内存开销。

> 架构直觉：在系统架构中，任何宣称“完美解决性能问题”的方案，背后一定隐藏着残酷的交易。
> 
> **堆外内存池用“降低容灾底线”和“翻倍的内存开销”为筹码，向操作系统赎回了对内存的绝对控制权**。这就是架构设计的迷人之处——永远没有银弹，只有权衡。

##### 读写分离架构：Write/Commit/Flush三阶段流转

**具体的工程问题**

* **工程问题一：堆外“私人金库”与内核 Page Cache 之间的物理鸿沟**
  * 在节点 2 中，我们利用 `TransientStorePool` 让业务线程把数据写到了用户态的堆外内存中。但数据最终必须落入磁盘，而通往磁盘的必经之路依然是操作系统的 Page Cache。
  * 如果让高并发的业务写线程自己去把数据从堆外内存拷贝到 Page Cache，那等于又把它们推向了与内核机制硬碰硬的深渊。如何构建一条平滑的运输线，**将堆外的数据“偷渡”进内核空间**？


* **工程问题二：消费者冷读引发的“同源内存踩踏”（读写争用）**
  * 在传统架构下，读和写都发生在 Mmap（Page Cache）中。当消费者进行落后极多的冷数据拉取时，大量沉睡在磁盘的历史数据会被 OS 疯狂换入 Page Cache，无情地将正在高频写入的热数据挤出内存。
  * 这种“读写同源”导致的内存踩踏，会引发极高的缺页中断和 I/O 阻塞。如何在物理内存层面实现真正的**读写隔离**？



---

**技术执行流**

* 为了彻底解决数据过渡与读写踩踏，RocketMQ 在开启 `TransientStorePool` 后，将原本两步走的 `Put -> Flush` 架构，重构为极其精密的 `Write -> Commit -> Flush` 三阶段异步流水线。
* **引入静态组件**：
  * **CommitRealTimeService（异步提交线程）**：负责将堆外内存数据搬运到内核态的守护线程。
  * **FlushRealTimeService（异步刷盘线程）**：负责将内核态脏页物理落盘的守护线程。
  * **FileChannel（文件通道）**：Java NIO 组件，用于执行底层的系统调用，将用户态内存数据写入 Page Cache。


* 三阶段流转与内存级读写分离的执行流：
  1. **阶段一：Write（极致写堆外，绝不阻塞）**：
     * 业务写线程将消息拼装好，完全以纯内存拷贝的方式，极速写入借来的 **DirectByteBuffer**（堆外内存）。
     * 更新 `wrotePosition`（写入游标），写线程立刻向 Producer 返回成功（ACK）。此时主链路结束，耗时极短。


  2. **阶段二：Commit（异步入内核，数据偷渡）**：
     * 后台 **CommitRealTimeService** 线程以固定频率（默认 200ms）醒来。
     * 线程调用 `FileChannel.write()`，将刚才写入 DirectByteBuffer 中的那一批脏数据，提交（搬运）到操作系统的 Page Cache 中。
     * *易错点：这一步本质上是一次 CPU 内存拷贝（从 JVM 用户态拷贝到 OS 内核态），数据尚未落盘，但已经交给了操作系统管理。*
     * 更新 `committedPosition`（提交游标）。


  3. **阶段三：Flush（异步落物理磁盘）**：
     * 另一根后台线程 **FlushRealTimeService** 周期性醒来，调用 `FileChannel.force()`。
     * 强迫 OS 将 Page Cache 中 `committedPosition` 范围内的新脏页刷入物理硬盘。更新 `flushedPosition`（刷盘游标）。


  4. **物理读写分离的达成（解决工程问题二）**：
     * 当消费者发起消息拉取请求时，引擎**永远只从 MappedByteBuffer（Page Cache）** 中读取数据。
     * 此时奇迹发生了：写链路的战火全部集中在 `DirectByteBuffer`（堆外），而读链路的战火全部集中在 `Page Cache`（内核）。消费者不管怎么疯狂回溯历史数据、怎么把 Page Cache 搅得天翻地覆，**都绝对波及不到主写链路所在的堆外内存**。





---

**底层原理**

1. **指令流水线（Pipeline）与 I/O 降维解耦**
* 借鉴 CPU 流水线思想，将一次沉重且充满不确定性的 I/O 操作，切分为三个纯粹的微观动作：极速的内存赋值（Write）、用户态到内核态的批处理拷贝（Commit）、内核态到磁盘的异步落盘（Flush）。不同阶段由不同的守护线程独立驱动，彻底解放了主业务线程。


2. **读写分离的物理降维（Read-Write Memory Isolation）**
* 传统数据库的读写分离通常指“主库写、从库读”。而 RocketMQ 利用 `TransientStorePool` 实现了极其硬核的**单机内存级读写分离**。用双倍物理内存的代价，换取了读写操作在 CPU 缓存和物理页表上的绝对无锁并发。



---

**架构决策与API映射**

* **对应工程问题一：提交频率与 CPU 算力的平衡**
* **决策思路**：Commit 操作本质上是 CPU 内存拷贝。如果每一条消息都立刻 Commit，会产生海量的系统调用，极度消耗 CPU。必须采用批处理的思想，攒够一定数量的脏页（默认 4 页即 16KB）才执行一次批量提交，以压低 CPU 占用率。


```java
// 【核心配置】触发 Commit 的最小页数。
// 默认 4 页（16KB）。如果这一批次积累的数据不足 4 页，本次后台轮询将跳过提交。
commitCommitLogLeastPages = 4

// 【核心配置】强制 Commit 的最大间隔。
// 防止在业务低谷期，数据迟迟凑不够 4 页而滞留在堆外内存。默认 200 毫秒强制提交一次。
commitCommitLogThoroughInterval = 200 

```



---

**边界与局限**

* **微小的“不可见幻读”窗口（消费延迟）**：
* 业务方明明收到了写成功的 ACK，但这其实只是 Write 到了堆外内存。消费者只能从 Page Cache 里读数据。
* 这意味着，在数据被后台线程 Commit 到 Page Cache 之前（最大间隔可能是 200ms），**消费者是绝对拉取不到这条消息的**。对于要求亚毫秒级极限端到端延迟的场景，这种三阶段架构反而会拖慢消费响应速度。


* **无意义的 CPU 内存拷贝税**：
* 不开启堆外内存时，Mmap 机制是“零拷贝”的（直接操作 Page Cache）。而开启堆外内存后，为了实现读写分离，系统被迫要在 `Commit` 阶段使用 CPU 将数据从 DirectByteBuffer 搬运到 Page Cache。在几十万 TPS 的极高吞吐下，这会成为一个不可忽视的 CPU 计算开销点。



---

**关键点**

* 开启 `TransientStorePool` 后，RocketMQ 的数据流转被硬性拆分为 `Write -> Commit -> Flush` 三阶段。
* 业务线程只负责写纯粹的堆外内存，实现了 $O(1)$ 的写延迟，且彻底摆脱了 OS 刷盘毛刺。
* 读操作被死死钉在 Page Cache 中，物理层面上达成了极其优雅的“写堆外，读内核”的单机读写分离。

> 架构直觉：在单机极限性能优化中，如果一段共享资源（Page Cache）无法避免高频的读写冲突，最暴力的解法就是“再造一个世界”。把写操作引流到纯粹的法外之地（堆外内存），用异步搬运工（Commit）去衔接，从而在物理隔离的废墟上重建极致的吞吐量。
