##### String

`String`不是一个简单的 `key-value` 缓存
- 每一次 `SET` 操作的背后，都是 **CPU 缓存行**与**内存碎片**之间的一场较量
- 在分布式高并发场景下，**哪怕是极其微小的一个 String 设计失误**，**都有可能直接卡死底层单线程的 EventLoop事件循环**，**导致整个微服务集群雪崩**。


---

`SET key value` 不仅仅是在内存里开辟了一块空间把数据塞进去。Redis 的作者 antirez 在设计 String 时，其实是在和 **CPU L1 Cache、jemalloc 内存分配器** 进行极其变态的博弈。


**C 语言原生字符串的物理缺陷**
- Redis 是用 C 语言写的，但 antirez 极其嫌弃 C 语言原生的字符串（以 `\0` 结尾的字符数组）。在单线程高并发的 Redis 面前，C 字符串简直就是性能毒药。

- **C 语言原生字符串的“三宗罪”：**
  1. **O(N) 的死等（获取长度）：** C 字符串本身不记录长度。你要想知道一个字符串有多长，CPU 必须从头开始遍历，直到遇到 `\0` 为止。如果这是一个 5MB 的 JSON，主线程就得干巴巴地跑一个 O(N) 的循环，这在单线程模型里是绝对不可容忍的阻塞！
  2. **二进制不安全（图片/视频的噩梦）：** C 语言遇到 `\0` 就认为字符串结束了。如果你缓存的是一段音频、图片转换成的二进制流，中间恰好包含 `\0` 的字节，C 字符串会直接将其**无情截断**。
  3. **内存重分配的极其昂贵：** 每次你想追加字符串（Append），C 语言都必须重新向操作系统申请一块更大的内存（`malloc`），再把老数据拷过去。**系统调用是极其昂贵的上下文切换！**

---

**SDS（简单动态字符串）**
- 为了彻底干掉上述痛点，Redis 自己造了一个数据结构：**SDS (Simple Dynamic String)**。
- 它的宏观结构极其粗暴但有效：**三个元数据+存放数据的数组**
  ```c
  struct __attribute__ ((__packed__)) sdshdr8 {
      uint8_t len;     // 已使用的字节数
      uint8_t alloc;   // 总共分配的字节数（不包含头部和尾部\0）
      unsigned char flags; // 3位类型，5位未使用
      char buf[];      // 真正存数据的柔性数组
  };
  ```

**SDS 是如何完成降维打击的？**
* **O(1) 闪电获取长度：** 直接读 `len` 字段，查长度的时间复杂度从 O(N) 暴降到 O(1)。
* **绝对的二进制安全：** 判断字符串结束不再依靠 `\0`，而是严格依靠 `len` 字段。你哪怕在 `buf` 里塞满 `\0`，SDS 照样能完整读出来。
* **核心防御：空间预分配与惰性释放（对抗系统调用）**
  * **预分配：** 
    * 如果你修改后 SDS 长度小于 1MB，Redis 会直接给你分配**双倍**的内存（`alloc = 2 * len`）；
    * 如果大于 1MB，则预留 1MB。下次再 Append 时，如果空间够，**绝对不触发底层 `malloc`**。
  * **惰性释放：** 字符串缩短时，并不立刻归还内存给操作系统，而是仅仅修改 `len` 的值，留着备用。**用极小的空间浪费，换取极致的 CPU 性能。**

---

**极致压榨：String 的三种变身形态与物理内存布局**
- Redis 在存入一个 String 时，外层还会包装一个超级对象头：`redisObject`（占用 16 字节）。为了极致压榨内存，Redis 会根据你输入的内容，在底层自动切换 **3 种编码格式（Encoding）**。
  - 在全局哈希字典中，**`*key`指针存储键名，`*val`指针存储`redisObject`**
- **`int` 编码：白嫖指针空间**
  * **触发条件：** 当你存的是一个纯数字（如 `SET count 100`），且可以用 8 字节的 `long` 表示时。
  * **物理布局：** `redisObject` 里本来有个 8 字节的 `ptr` 指针（指向实际数据）。既然存的是数字，Redis 直接把这 8 字节指针**当成数字本身**来用！
  * **宏观认知：** **0 额外内存开销**，甚至连 SDS 都不用创建。这就是为什么在 Redis 里做计数器（`INCR`）性能高得离谱的物理原因。


- **`embstr` 编码：连续内存的艺术**（极限卡位 44 字节）
  * **触发条件：** 存的是字符串，且长度 **≤ 44 字节**。
  * **物理布局（极度高能）：**
    * 普通的对象需要两次 `malloc`（一次给 `redisObject`，一次给 `SDS`）。
    * `embstr` 极度狡猾，它只调用 **1 次 `malloc`**，向操作系统申请一块**绝对连续**的内存空间，把 `redisObject` 和 `SDS` 紧紧凑在一起。
  * **为什么偏偏是 44 字节？（架构师必考题）**
    * 现代 CPU 读取内存不是一个字节一个字节读的，而是每次读取一个 **Cache Line（缓存行，通常是 64 字节）**。
    * 内存分配器 jemalloc 分配内存时，为了减少碎片，习惯分配 64 字节的块。
    * `redisObject` 占 16 字节，SDS 头（`sdshdr8`）占 3 字节，加上结尾的 `\0` 占 1 字节。
    * **64 - 16 - 3 - 1 = 44 字节！**
  * **宏观认知：** 只要你的小字符串不超过 44 字节，它就能被 CPU 一次性全部吸入 L1 Cache 中，**缓存命中率极高，性能爆表！**


- **`raw` 编码：指针跳跃的灾难**
  * **触发条件：** 字符串长度 **> 44 字节**。
  * **物理布局：** 调用 **2 次 `malloc`**。`redisObject` 在一块内存，`SDS` 在另一块内存，中间靠指针连接。
  * **宏观认知：** 内存变得不连续。CPU 读取时大概率会发生 Cache Miss。当你的并发量达到 10万 QPS 时，这细微的指针跳转和缓存未命中，就会成为性能瓶颈。

### 四、 架构师的噩梦：微服务下的内存碎片陷阱

**陷阱：** 如果你用 `SET` 存了一个 20 字节的字符串（此时是 `embstr`），然后业务逻辑调用了一次 `APPEND` 命令往后加了几个字符（总长度仍未超过 44 字节），底层会发生什么？

**物理推演：**
`embstr` 是**只读**的（因为它是一块连续的内存，紧凑得没有任何预留空间）。只要你对 `embstr` 执行了哪怕 1 字节的修改，Redis 也会**立刻将其撕裂**，重新申请两块内存，强行转换成 `raw` 编码！
这就导致：原本完美贴合 64 字节 Cache Line 的内存被抛弃，产生了内存碎片；同时触发了昂贵的系统调用和数据搬运。

> **架构指导原则：**
> 在微服务中，**不要把 Redis String 当成可以随意拼接的 StringBuffer！** 如果你要存频繁修改的结构化数据，请老老实实去用 Hash 结构，或者在后端代码里拼接好 JSON，一次性 `SET` 进去！
